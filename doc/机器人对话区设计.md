# 机器人对话区设计（统一对话池 + 多会话切换，全渠道）

> 目标：把这个 AI 的所有对话——网页端发起的、以及各平台机器人（QQ / 飞书 / 后续任何平台）产生的——
> 统一放进**一个"机器人对话区"**。
>
> 区内**不分平台、不分用户、不做隔离**：任何人（网页主人或任意平台的外部用户）通过 AI 都能
> **列出 / 切换 / 新建**这个区里的**全部**对话。投递仍跟着"人"走，天然支持并发，无需任何锁。
>
> **适用全渠道**：QQ、飞书及后续接入的任何平台，产生的会话都落进同一个对话区。
>
> 状态：设计稿（尚未实现）。分支：`claude/doc-edit-conversation-design-qFY8k`。

---

## 一、问题与背景澄清

### 1.1 现象
服务器联调 QQ 机器人：能连上、能收到用户消息，但"发不回去"，或感觉只能跟一个对话绑定。

### 1.2 核心误解
QQ 官方机器人**没有"同一时刻只能锁定一个对话"的限制**。所谓"锁定"是对**被动回复机制**的误读：

- **被动回复**：用收到的那条用户消息的 `msg_id`（或事件 `event_id`）作为凭证回复。
  - 有效期 **5 分钟**；
  - 同一个 `msg_id` 下可回复多条，用 `msg_seq` 区分（从 1 开始，**同一 msg_id 内必须唯一递增**）。
- **主动推送**：不带 `msg_id` 直接推，需要单独申请的主动消息权限，且有严格频控；没权限会被拒。

关键结论：

> **投递地址跟着"人"走（openid + msg_id），会话只是"消息归到哪段历史"的逻辑分组。二者完全独立。**
> 机器人永远是"回复刚才给我发消息的那个人"，不管这条消息归到哪个 `session_id`，都能正常发回去。
> 因此"切换会话"和"能不能回复"互不冲突 —— **不存在锁**。

### 1.3 当前代码为什么"显得"被锁死
`server/connector_runtime/bots/qq/router.py` 把会话写死了：

```python
session_key = f"{target_type}_{target_id}"
session_id  = f"qq_{config_id}_{session_key}"   # 一个 openid = 永远唯一一条会话
```

一个 openid 永远映射到同一个 `session_id`，所以这个 QQ 用户无法切到该 AI 的其它对话记录。
这只是**路由策略写死了**，不是平台限制。

## 二、产品决策（已确认）

| 项 | 决策 |
| --- | --- |
| 对话区划分 | **不分区**。网页 + 所有平台 + 所有用户的会话，统一进**同一个机器人对话区**。 |
| 隔离 | **不隔离**。任何用户（网页主人 / 任意平台外部用户）都能列出 / 切换该 AI 的**全部**对话（含网页端及他人建的）。 |
| 多会话切换 | **保留**。由 AI 调 MCP 工具完成（用户自然语言说"换个对话 / 新开一个 / 列一下"，AI 理解后调工具），不用前缀指令。 |
| MCP 跨进程上下文 | **打通 HTTP 上下文**（在 `/internal/mcp/call` 传递 `session_context`）。 |

> 与上一版设计的区别：**删除**了"按平台 / 按用户分对话区（栏目化）"和"按身份隔离"两套机制。
> 现在只有"一个 AI = 一个共享对话池"，多会话切换作用于整个池。

## 三、关键架构事实（实现前必须知道）

### 3.1 部署是分进程的（remote 模式）
`docker-compose.yml`：

```
MCP_RUNTIME_URL=http://mcp-runtime:3001
CONNECTOR_RUNTIME_URL=http://connector-runtime:3002
AI_RUNTIME_URL=http://ai-runtime:3003
AI_DISPATCH_MODE=remote
```

含义：**MCP 工具是跨进程 HTTP 调用的**。
`server/ai_runtime/inference/core.py::_call_mcp_via_runtime` 只在 body 里发
`{tool, user_id, ai_config_id, arguments}`，**不带 `session_id`**。

因此工具进程里 `get_run_session_context()` **返回 None**。
→ 现有 `conversation.forget_before_current / find / create / delete` 在该部署下其实是
**降级依赖 AI 自己传 `session_id`**；要让切换 / 列出 / 新建工具可靠知道"当前是哪条会话"，
**必须把会话上下文打通到工具进程**。

### 3.2 会话上下文当前在哪设置
`server/ai_runtime/inference/core.py` worker 内：

```python
set_run_session_context({
    "user_id": user_id,
    "ai_config_id": ai_config_id,
    "ai_kind": ai_kind,
    "session_id": session_id,
    "session_name": session_name,
    "model": model,
    "current_user_message_id": current_user_message_id,
})
```

这是 `connector_runtime.dispatch.agent_dispatch` 里的 `contextvar`，**仅在 worker 进程 / 线程内有效**，
跨 HTTP 到 mcp-runtime 后失效。

### 3.3 收发链路（现状）
```
long_connection.py  (botpy 长连接收消息)
        │  on_c2c_message_create / on_group_at_message_create ...
        ▼
router.py::handle_qq_event_payload
        │  落库 user 消息 + 起 ChatRun + register_qq_session_route
        ▼
ai_runtime worker (core._run_worker) 生成回复，落库 assistant 消息
        │  chat_persistence._save_message → notify.notify_saved_assistant_message
        ▼
notify.py  → bot.load_session_route(session_id) → 拿到 target_id(openid)/msg_id/msg_seq
        ▼
qq/adapter.py::notify_assistant_message → service.send_qq_text_message (被动回复)
```

路由表 `BotSessionRoute` 按 `(channel, user_id, ai_config_id, ai_kind, session_id)` 唯一，
`target_json` 存 `{target_id, target_type}`，外加 QQ 专用列 `source_message_id / source_event_id / next_msg_seq`。

## 四、设计方案

### 4.1 核心思想
**一个 AI = 一个共享对话池（"机器人对话区"）。**

- **投递不变**：仍按 `session_id → BotSessionRoute → openid + msg_id` 被动回复，天然支持并发多会话，无需任何锁。
- **活跃会话游标**：每个入站身份（identity）有一个"当前活跃会话"指针，决定它的下一条消息落到池里哪条会话。
  无游标时落到该身份的默认 home 会话（向后兼容）。游标可指向**池中任意会话**（无同区 / 同身份限制）。
- **不隔离**：`list / switch / new` 作用于该 AI 的**整个对话池**，不按 channel、不按身份过滤。
  即任何用户都能看到并切到网页端及他人建的会话。

> 为什么仍需"按身份的游标"而不是全局唯一游标：多个外部用户可能同时跟同一个 AI 说话，
> 每个人的消息需要有确定的落点；用全局游标会让 A 的切换影响 B 的消息归属，产生串话。
> 因此**游标按身份记录（决定"我这条消息写到哪")，但可指向的范围 = 整个池（不隔离）。**

### 4.2 切换语义
- AI 在对话中调 `conversation.switch`：**当前这条消息仍在原会话回复**（回复跟着 openid 走，照常发回），
  **下一条用户消息起进入新会话**，AI 回一句确认。
- 新建 `conversation.new`：建空会话 + 登记路由（写一条 `BotSessionRoute` 占位行，`source_message_id` 留空，
  下一条入站消息会补上）+ 设游标，下一条起进入新会话。

## 五、改动清单（5 项）

### 改动 1：打通跨进程会话上下文（基础，顺带修好现有工具）
- `server/ai_runtime/inference/core.py::_call_mcp_via_runtime`
  在 HTTP body 增加 `session_context`（即当前 run 的 `get_run_session_context()` 快照）。
- `server/mcp_runtime/app.py` 的 `CallRequest` 增加可选 `session_context` 字段；
  `POST /internal/mcp/call` 收到后用 `set_run_session_context(...)` 包裹 `registry.call`，调用结束复位。
- 附带收益：现有 `conversation.forget_before_current / find / create / delete` 在 remote 模式下
  也能可靠拿到 `session_id`，不再依赖 AI 猜参数。

> 注意：in-process 模式（`MCP_RUNTIME_URL` 为空）走 `registry.call`，contextvar 本就有效；
> 此改动只影响 remote HTTP 路径，需保持两种模式都正确。

### 改动 2：新增活跃会话游标表 `BotUserCursor`
位置：`server/api/models/bot_session_route.py`，并在 `server/api/models/__init__.py` 导出。

```python
class BotUserCursor(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    channel: str = Field(index=True)
    user_id: int = Field(foreign_key="user.id", index=True)   # AI 拥有者
    ai_config_id: int = Field(foreign_key="assistantaiconfig.id", index=True)
    ai_kind: str = Field(default="core", index=True)
    identity_key: str = Field(index=True)                      # 身份键（QQ openid / 飞书 receive_id / …）
    active_session_id: str = Field(default="")                 # 游标：该身份当前活跃会话
    created_at: float = Field(default_factory=time.time)
    updated_at: float = Field(default_factory=time.time)
```

> 新表由各 runtime 启动时的 `create_db_and_tables()`（`SQLModel.metadata.create_all`）自动创建，
> **无需写 ALTER 迁移**（`create_all` 只建缺失的表，不改已有表）。

唯一性：逻辑上 `(channel, ai_config_id, ai_kind, identity_key)` 唯一，读写时按这四元组 upsert。

> 相比上一版：**去掉**了 `alias / display_name` 等字段——不做对话区命名 / 分组，纯粹只是个游标。

### 改动 3：游标 helper（**渠道无关**，新建共享模块）
位置：新建 `server/connector_runtime/bots/session_cursor.py`，所有 bot 共用；
**不放在 `qq/` 下**，因为飞书及后续平台都要走同一套。

所有 helper 以**通用身份键 `identity_key`** 为单位（不写 `openid`）：
- QQ：`identity_key = target_id`（openid）
- 飞书：`identity_key = receive_id`（chat_id 或 open_id）
- 身份键由各 `BotAdapter.route_identity_key(target_json)` 提供（见改动 4）。

- `get_active_session_id(session, *, channel, user_id, ai_config_id, ai_kind, identity_key, default) -> str`
  - 查 `BotUserCursor`；若指向的会话已不存在，回退 `default` 并重置游标。
  - **不校验会话归属**（不隔离）：游标允许指向池中任意会话。
- `set_active_session_id(session, *, channel, user_id, ai_config_id, ai_kind, identity_key, session_id)`
  - upsert 游标。
- `list_ai_sessions(session, *, user_id, ai_config_id, ai_kind) -> list[dict]`
  - 列出该 AI 的**全部** `ChatSession`（网页 + 各平台机器人，**不按 channel / 身份过滤**）；
    标注每条的来源（web / 各 channel）与是否为当前身份的活跃会话，供 `conversation.list` 使用。

### 改动 4：适配器契约 + 入站路由按游标解析
**适配器**：在 `server/connector_runtime/bots/base.py` 给 `BotAdapter` 增加：

```python
def route_identity_key(self, target_json: dict) -> str:
    """从本渠道寻址载荷里解出身份键。QQ→target_id；飞书→receive_id。"""
```
各 adapter（`qq/adapter.py`、`feishu/adapter.py`、未来平台）各自实现。

**入站路由**：`qq/router.py::handle_qq_event_payload` 与
`feishu/router.py::handle_feishu_event_payload` 把写死的 `session_id` 改为调用共享 helper：

```python
# QQ:    identity_key = target_id;        home = f"qq_{config_id}_{target_type}_{target_id}"
# 飞书:  identity_key = chat_id|open_id;   home = f"feishu_{config_id}_{chat_id|open_id}"
session_id = get_active_session_id(
    session, channel=<bot.channel>,
    user_id=cfg.user_id, ai_config_id=cfg.id, ai_kind=ai_kind,
    identity_key=identity_key, default=home_session_id,
)
# 首次接触：游标不存在时落为 home，并确保 home 的归属路由存在
```
`session_name` 优先取该 `session_id` 对应 `ChatSession.session_name`，否则默认名。
其余（落库、`register_*_session_route`、起 run、busy 回复、deferred）逻辑不变。

> 两个 router 现状高度对称，可顺手把公共入站流程抽到共享函数（可选优化，非必须）。

### 改动 5：三个 MCP 工具（**天然渠道无关，作用于整个对话池**）
位置：handler 写在 `server/mcp_runtime/mcp/tools/conversation.py`；
注册在 `server/mcp_runtime/mcp/registry.py`；权限在 `server/mcp_runtime/mcp/permissions.py`（均 `ROLE_MEMBER`）。

所有工具先用 `get_run_session_context()`（改动 1 后跨进程可用）拿到当前 `session_id` / `channel` / `identity_key`，
据此设置游标；列举 / 切换的**范围是该 AI 的整个对话池，不做任何过滤**。

- `conversation.list`
  - 列出该 AI 的全部会话（id / 名称 / 来源 channel / 最近消息时间 / 是否当前活跃）。
- `conversation.switch`
  - 入参 `session_id`（或 `name` / `query` 解析）；目标会话只要属于该 AI 即可（**不校验 channel / 身份**）；
    `set_active_session_id`；返回"已切换，下一条消息起生效"。
- `conversation.new`
  - 入参 `name`；建 `ChatSession` + 写 `BotSessionRoute` 路由占位行（复制当前 `target_json`）+ 设游标。
  - 与现有 `conversation.create` 区别：`new` 会把当前身份的游标切过去；`create` 保持通用不动。

### 改动 6：各渠道运行时提示词加引导
在 `qq/router.py::_build_qq_runtime_prompt` 与 `feishu/router.py::_build_feishu_runtime_prompt`（及后续平台同名构造）追加：
当用户表达"换 / 切换 / 列出 / 新开 对话 / 会话"时，调用 `conversation.list / switch / new`；
切换对下一条消息生效，本条回复照常发回当前对话。

## 六、边界与注意事项

1. **向后兼容**：老用户（QQ / 飞书）无游标 → 自动用 home 会话，行为与现状一致。
2. **活跃会话被删**：游标指向不存在的会话时，回退 home 并重置游标，避免"发不出 / 找不到"。
3. **不隔离是有意为之**：`list / switch` **不**按身份或平台过滤——任何外部用户都能看到并切到
   网页端及其他用户建的对话。**这意味着外部用户能读到网页端的私密对话**，已与产品方确认接受此行为。
4. **并发落点**：游标按身份记录，避免多人同时说话时互相串话；但多人若都切到同一会话，消息会交织在一起（符合"共享池"预期）。
5. **被动窗口 / msg_seq**（与本特性正交，但建议一并体检）：
   - `routes_store.py` 里 `next_msg_seq` 跨 `msg_id` 只增不重置 —— 新 `msg_id` 应从 `seq=1` 重新开始；
   - AI 生成慢或被 `_wait_for_qq_idle_then_run`（最长可等 24h）拖到 5 分钟外，`msg_id` 过期 → 被动回复必失败，
     目前**无降级、错误被静默吞掉**（只 `logger.exception`）。
     建议：发送失败时把腾讯返回的 `code/message` 透出到日志与 `/api/qq/diagnose`，便于定位。
6. **两种部署模式都要测**：`MCP_RUNTIME_URL` 置空（in-process）与设值（remote HTTP）下，切换工具都要正确。

## 七、实现顺序建议

1. 改动 1（HTTP 上下文）→ 单独验证现有 `conversation.find` 在 remote 模式能拿到 `session_id`。
2. 改动 2 + 3（游标表 + helper）。
3. 改动 4（适配器 `route_identity_key` + 路由按游标解析，默认 home，确保兼容）。
4. 改动 5（三个工具）+ 6（prompt）。
5. 端到端联调：QQ / 飞书里说"列出对话""新开一个聊旅游的""切回刚才那个""切到网页上那条"，
   验证整个池可见可切、回复投递正常。

> 落地建议：**先 QQ 跑通 → 再把飞书 router 切到共享 helper（结构对称，改动极小）→ 之后任何新平台自动复用**。
