## 详细条目

### 1. 大文件拆分

**痛点**：单文件巨石让 review、跳转、merge conflict 都很痛苦。

- `server/api/routers/chat_worker.py` — 84KB / 1900+ 行
- `server/api/services/librarian_service.py` — 57KB
- `server/api/services/ai_message_service.py` — 38KB
- `server/api/services/valhalla_service.py` — 24KB
- `server/api/routers/chat_action_routes.py` — 23KB
- `server/api/routers/chat_prompt_utils.py` — 26KB
- `server/api/routers/chat_scheduler.py` — 18KB
- `server/api/routers/chat_stream.py` — 18KB

**目标**：每个文件 ≤ 600 行，按职责切分。`chat_worker.py` 推荐切法：
- `chat_worker/dispatcher.py` —— 取队列、claim run、并发槽位
- `chat_worker/runner.py` —— 实际 LLM 调用主循环
- `chat_worker/tool_orchestrator.py` —— MCP 工具路由（registry → mcp-runtime HTTP fallback）
- `chat_worker/context.py` —— 上下文裁剪、token 预算、prompt 组装入口
- `chat_worker/persistence.py` —— 消息落库 + 状态推送

**交付**：每个大文件落地新目录后，旧文件仅作 re-export shim 一段时间，再彻底删除。

### 2. Pydantic Settings 收编环境变量

**痛点**：`os.environ.get("HEYSURE_X")` 散落十几处，命名拼写靠 grep 验证，没有默认值集中说明。

**当前散布点**：
- `api/core/config.py`
- `gateway/app.py`、`gateway/main.py`
- `ai_runtime/main.py`、`ai_runtime/worker.py`
- `mcp_runtime/main.py`、`connector_runtime/main.py`
- `api/sio.py`
- `api/runtime/internal_http.py`
- `api/routers/chat_action_routes.py`（`AI_DISPATCH_MODE`）
- 等等

**目标**：`api/core/settings.py` 一个 `Settings(BaseSettings)` 全收：

```python
class Settings(BaseSettings):
    service_role: Literal["gateway", "worker", "mcp", "connector"] = "gateway"
    database_url: str
    internal_token: str = ""
    api_gateway_url: str = ""
    mcp_runtime_url: str = ""
    connector_runtime_url: str = ""
    server_reload: bool = False
    ai_dispatch_mode: Literal["local", "remote"] = "local"
    ai_runtime_max_concurrent: int = 16
    ai_runtime_port: int = 3001
    connector_runtime_port: int = 3002
    ai_debug: bool = False
    class Config:
        env_prefix = "HEYSURE_"
        env_file = ".env"
```

调用方改成 `from api.core.settings import settings`。

**交付**：单一 settings 对象、一次启动校验、`.env.example` 自动从字段生成、文档更新。

### 3. `print()` → `logging`

**痛点**：生产容器里 `docker logs` 一片混乱，没有 level、没有 module、没法 grep 过滤。

**当前统计**：server/ 下 `print(` 调用约 200+ 处，主要在：
- `gateway/app.py`（lifespan + 路由发现）
- `connector_runtime/app.py`（keepalive + 长连接生命周期）
- `ai_runtime/worker.py`（NOTIFY / claim / retry tick）
- `api/bots/*/long_connection.py`
- `api/services/agent_dispatch.py`
- `api/routers/chat_*.py`

**目标**：
- 每个模块顶部 `logger = logging.getLogger(__name__)`
- `print(f"[xxx] yyy")` → `logger.info("yyy")` / `logger.warning(...)` / `logger.error(...)`
- 启动时配置一次：dev 用 console formatter，prod 用 JSON formatter（方便 stack）
- `HEYSURE_LOG_LEVEL` 控制全局级别

**交付**：`api/core/logging_config.py` 一次性 setup，所有 `print` 替换完毕，docker 日志可直接喂给 grafana / loki。

### 4. Alembic 替代手写迁移

**痛点**：
- `api/core/migrations.py` 是 sqlite3 + 手写 `_add_column` 的拼凑
- 我们刚加的 `run_data_consolidations` 又走 SQLAlchemy engine，两套并存
- Postgres 部署的迁移本质上靠 `create_all` 或一次性脚本，没有可重复路径
- 多人合作时容易出现"先合的 PR 加了列，后合的 PR 又删"的怪状

**目标**：
- `alembic init server/alembic`
- 把现有 schema 生成 baseline revision
- `_consolidate_*` 改写成 Alembic data migration
- `database.py.create_db_and_tables` 改为 `alembic upgrade head`

**交付**：版本号 + downgrade 路径、CI 检查 `alembic check`、迁移文档。

**风险**：现有 sqlite/postgres 都得跑一次 baseline 标记，部署时要协调。

### 5. 前端 dashboard 数据类型化

**痛点**：
- `web/src/api/ai.ts` 里 `AiCardRow = Record<string, any>`、`AiConfigRow = Record<string, any>`
- 模板里 `card.feishu_status`、`card.bot_configs.feishu.app_id` 全靠 ts-ignore 的等价物
- 我们刚把 `bot_configs` 类型化了，其他字段还散在各处

**目标**：在 `web/src/types/api.ts` 补全：

```typescript
export interface AiCardRow {
  id: number
  name: string
  bot_channel: 'feishu' | 'qq'
  bot_configs: BotConfigs
  bot_statuses: Record<string, BotStatus>
  // ...其余字段也补完
}
```

**交付**：消除 `Record<string, any>`、组件 props 改成 typed、watcher 编译期能查 path 错。

### 6. `BotAdapter.diagnose()` 抽象 + 统一诊断端点

**痛点**：
- 现在只有 QQ 有 `/api/qq/diagnose/{id}`（验 app_id / app_secret / token / target）
- 飞书没有，加新机器人也得各自再写一份

**目标**：
- `BotAdapter.diagnose(cfg) -> Dict[str, Any]`（默认实现返回 `{"supported": False}`）
- 网关挂一个 `/api/bots/{channel}/diagnose/{cfg_id}`，按注册表分发
- QQ 现有的 `/api/qq/diagnose/{id}` 改为 shim → `/api/bots/qq/diagnose/{id}`，半年后下线

**交付**：每家机器人配置生效与否的"自检按钮"在前端长一个样子。

### 7. 自动化测试 baseline

**痛点**：项目里没看到 `tests/` 目录或 `pytest.ini`，回归全靠手动跑。我们刚做的两次重构都没有测试托底。

**目标**（建议从 bot adapter 起步，接口窄、副作用清晰）：
- `tests/bots/test_config_store.py` —— `load / save / get_channel_config / update_channel_config` 的纯函数测试
- `tests/bots/test_registry.py` —— 注册顺序、`iter_active_for_config`、`channel_for_session_id`
- `tests/bots/test_feishu_adapter.py` —— `is_enabled`、`build_status`、`default_config` 的最小快照
- `tests/bots/test_routes_store.py` —— `BotSessionRoute` 的 upsert + view 解码（用 sqlite memory）
- CI 用 GitHub Actions / 现有的 runner 跑 `pytest -q`

**交付**：`tests/` 目录 + pytest 配置 + 第一批 ~20 条用例，覆盖 bot 包。

### 8. 拆 `AiConfigModal.vue` 大组件

**痛点**：单 Vue 文件 1000+ 行，所有 tab 的表单挤在一起，改一处影响全局。

**目标**：按 tab 拆子组件：
- `<AiConfigBasicTab :form />`
- `<AiConfigBotTab :form />`（飞书 / QQ 表单，未来加新机器人就长一个子组件）
- `<AiConfigMcpTab :form />`
- `<AiConfigPromptTab :form />`
- `<AiConfigAutoControlTab :form />`

**交付**：每个子组件 ≤ 200 行；新机器人只动 `<AiConfigBotTab>` 一个文件。

## 推荐推进顺序

第一波（低风险、即时收益）：
- **#2 Settings** + **#3 logging** 一起做，同一改造的两面，可一周内拿下。
- **#6 diagnose 抽象** —— 顺便补完上一轮 bot 重构的最后一块。

第二波（中风险、需要协调）：
- **#5 前端类型化** —— 单独成 PR，不影响后端。
- **#7 测试 baseline** —— 趁刚重构完 bot 包记忆还热。

第三波（大动作、需要排期）：
- **#1 大文件拆分** —— `chat_worker.py` 先动；每次一个模块，分多 PR。
- **#4 Alembic** —— 需要一次集中协调，建议在大文件拆分完成后再启动。
- **#8 Vue 大组件** —— 可以和前端类型化合并。

## 度量

- 大文件指标：`find server -name "*.py" -exec wc -l {} + | sort -rn | head` 持续 < 600 行
- `print(` grep 数：从当前 ~200 降到 0
- 测试覆盖率：从 0% 起跑，先稳定 bot 包 > 80%
- 启动时长 / 迁移耗时：Alembic 切换后不应回退
