# 任务系统重构与流程沉淀设计

> 目标：把"任务"从**独立后台 job + 独立会话 + 独立面板**的重模型，重构为**对话内的一段可标记区间（span）**，
> 并在其上长出"摸索 → 固定流程 → 自动化"的三阶段流程沉淀能力。
>
> 本文只管**任务的交互模型与三阶段流程沉淀**。沉淀后的"知识/工具晋升、安全回归、度量淘汰"那套机器，
> 已在 [`进化流程设计.md`](进化流程设计.md) 讲透，本文直接复用、不重复。角色与知识流转见 [`角色与知识流转.md`](角色与知识流转.md)。
>
> ⚠️ 传承/英灵殿/generation 换代已废弃（见 `Agent进化与实战区域设计方案.md` 顶部声明），本文不依赖换代机制。

## 1. 一句话目标

让用户在对话里**自然地开启一个任务**，AI 以"先立带验收判据的阶段性目标、逐目标推进"的方式执行；
整段过程（目标 + 用到的工具名/参数 + 意外与注意事项）**自动结构化沉淀**为可复用流程；
随复用次数增加，流程从"每步都思考"逐步"固化"到"无脑串联、出错才接管"。

## 2. 核心洞察：先把"两类任务"分开（这是当前"乱"的根因）

当前实现把**前台任务**和**后台任务**用同一套"后台 job 机器"承载，于是前台任务也被套上了
`独立 session + 信件回执 + 新建表单`三件套，显得又重又乱。两者本质不同，应在概念上分开：

| 维度 | **前台任务**（用户在场，对话内发起） | **后台任务**（定时 / 无人值守） |
| --- | --- | --- |
| 形态 | 当前对话里的一段 **span**（`task.start` ~ `task.complete` 括起） | 独立 `AITaskJob` + 独立 run |
| 独立 session | **不需要**（就在当前会话里） | 需要（没人看着也得跑） |
| 信件回执 | **不需要**（结果就是对话下一条消息） | **保留**：回执本就是注入回创建者会话的系统消息，非独立信箱，不算冗余 |
| 触发方式 | AI 识别"像多步任务"→ 提议进入任务模式（轻确认）→ `task.start` | 调度器按 schedule 自动 `_start_task_run` |
| 新建表单 UI | **不需要**（对话里建） | 可砍大半，瘦成只读总览 |

> 一句话：**前台任务砍掉"独立 session / 信件 / 新建表单"，后台任务保留"独立 run / 回执"但栏目瘦身。**
> 这样"嵌入对话"与"无人值守仍可沉淀"两个诉求同时满足，互不打架。

## 3. 现状盘点：已有地基 vs 冗余

### 3.1 已有地基（别重造）

| 你设想的能力 | 现有实现位置 |
| --- | --- |
| 创建/管理任务的 MCP | `server/mcp_runtime/mcp/tools/tasks.py`：`task.create / update / list / complete` |
| 完成时要总结 | `task.complete(summary)` 强制 summary，并追写 `{workspace}/task.md` |
| 完成后回到原对话 | `server/api/services/task_completion_notify.py:notify_task_completion()`：回执追加进**创建者会话** |
| 工具调用已被逐条记录 | 每次调用存成 `ChatMessage`（tag=`mcp_tool_call`，含工具名/参数/结果），`ai_runtime/inference/core.py:_save_mcp_tool_bubble()` |
| 任务前注入相关知识 | `chat_scheduler.py` 派发时调 `librarian.brief()` 按触发词注入相关条目 |
| 流程/技能沉淀载体 | `librarian_service.py` propose→审批→`topics/*.md`；外加 `KnowledgeBase/inheritance_thoughts/**/SKILL.md` |
| 定时调度 | `chat_scheduler.py:process_task_scheduler / _ensure_scheduled_jobs`；schedule 规范 `services/task_schedule.py` |

### 3.2 冗余/缺口（本文要改的）

| 现状 | 问题 | 方向 |
| --- | --- | --- |
| 前台任务也走独立 `session_task_{job_id}` | 与"嵌入对话"诉求冲突，且割裂上下文 | 前台任务改为对话内 span，不新建 session |
| `TaskManagementModal.vue` 含重型新建表单 | AI 能在对话里建，表单大半冗余 | 瘦成"只读总览 + 暂停/删除" |
| `task.complete` 只存一句自然语言 summary | 无法复现"目标→工具→参数"轨迹，沉淀不可复用 | 完成时从 `mcp_tool_call` 气泡**结构化抽取**轨迹 |
| 无"关键词/意图触发"路径 | 只能 AI 显式调 `task.create` | 新增 `task.start`，由意图识别+轻确认驱动 |
| 阶段性目标无"成功判据" | 无法判定一步是否真成功（静默失败） | 目标必带 `success_criteria`（见 §5） |

## 4. 新交互模型：任务 = 对话里的一段 span

```
用户对话流（同一个 session，不分叉）
 … 普通聊天 …
 │
 ├─ AI 判断「像一个多步任务」→ 提议「要不要进入任务模式？」（轻确认，可一句话同意）
 │
 ├─ task.start(goal, success_criteria)        ← span 起点：立第一个带判据的阶段目标
 │     ├─ （后台静默采集本段内每次 mcp_tool_call 的 工具名/参数/出参/成败）
 │     ├─ 推进到目标 → 校验 success_criteria → 通过则立下一个阶段目标
 │     └─ 出意外 → AI 口头说明「发生了什么 / 注意事项」并当场调整（任务内短回路，不落库）
 │
 └─ task.complete(summary)                     ← span 终点：抽取整段轨迹 → propose 沉淀
       结果直接是对话里的下一条消息（前台无需信件回执）
```

要点：
- **span 不等于 session**：同一会话里用起止标记框住"任务这一段"，既嵌入对话又有干净的可抽取边界。
- **AI 只需口头汇报"意外/注意事项/大致流程"**；"目标 + 工具名 + 参数"由后端从气泡结构化采集，不靠 AI 凭记忆复述（与 `进化流程设计.md` P4 一致）。
- 后台定时任务沿用现有 `_start_task_run`（独立 run + 回执），但其内部同样用 `task.start/complete` 框 span，**共享同一套采集与沉淀逻辑**。

## 5. 三阶段 = 同一条流程的"成熟度等级"（不是三套机制）

你的"摸索 → 固定流程 → 自动化"，本质是同一条流程沿**"LLM 参与度递减、速度/成本递增、鲁棒性递减"**这条轴往下滑。
所以不做三套系统，而做**一条流程的三个成熟度等级**，并设"毕业门槛"防止不稳的流程滑进高风险等级。

| 阶段 | 成熟度 | LLM 参与 | 复用形态 | 进入下一级的门槛 |
| --- | --- | --- | --- | --- |
| 一：摸索与尝试 | `draft` | 全程推理 | 仅记录轨迹，未复用 | 同类任务被识别并成功复用 ≥1 次 |
| 二：固定流程测试 | `soft`（软复用） | 读 `SKILL.md` 重新解释执行 | 自然语言 SOP，每步自带判据；出错自动校正并回写 | 连续成功 ≥N 次 **且** 每步判据稳定 |
| 三：自动化调试 | `hard`（硬复用） | 仅出错才接管 | 带变量绑定的确定性串联，少/无中途思考 | —（终态，可因成功率下降被降级回 soft） |

> **默认软复用，硬复用是"毕业"才解锁。** 软复用抗环境漂移、抗参数变化；硬复用快但脆。
> 只有"连续成功 + 每步判据稳定"的流程才允许升 hard 进入阶段三这个雷区；一旦成功率掉下来，自动降级回 soft。
> 晋升/降级/淘汰的信号与阈值复用 `进化流程设计.md` 第 4 节度量与 P1 晋升规则。

### 5.1 三个硬约束（决定能不能落地，缺一不可）

1. **参数泛化**：轨迹里记的不能是上次的具体值（某路径/某 id），否则下次套不上。
   软复用阶段参数以自然语言"指代"留存（"用户刚提到的那个文件"）；升 hard 时才把可泛化的参数绑成变量。**记录易，参数化难，这是复用的命门。**
2. **复用匹配**：怎么判定"这次任务 == 过去某条流程"。起步复用 `librarian.brief()` 的触发词命中（粗但能用），
   误匹配宁可不复用也不要错复用——错复用比不复用更糟。
3. **成功校验**：阶段三"出错才接管"的前提是**错误可被观测**。但大量失败是静默的（结果错但无异常），
   所以**每个阶段性目标必须自带 `success_criteria`**——这也是 §4 把判据做成 `task.start` 必填项的原因。无判据则该流程永远不许升 hard。

## 6. UI 简化建议（直接回答"显示太复杂"）

| 现有 UI | 处置 | 理由 |
| --- | --- | --- |
| `TaskManagementModal` 的**新建任务表单** | **砍**（前台）/ 仅保留定时配置入口（后台） | 前台任务在对话里建；只有"定时/循环"这类无人值守配置才需要表单 |
| 任务**列表/总览** | **留并瘦身**：只读看板 + 暂停/删除/查看回放 | 后台任务需要一个"控制面板"知道有哪些定时任务、能叫停 |
| 任务**详情/对话回放** | 留 | 排障与复盘有价值 |
| **信件回执**（绿色系统消息） | 前台**不产生**；后台**保留** | 前台结果即对话下一条；后台无人值守才需异步回执 |
| 对话内**任务态指示** | **新增**（轻量）：当前在某任务 span 内时，显示"任务中 + 当前阶段目标"的内联条 | 让"进入了任务模式"这件事在对话里看得见，不靠独立面板 |

> 净效果：用户感知从"去任务栏目填表 → 任务在别处跑 → 回信箱看回执"三段式，
> 收敛为"在对话里说要做什么 → 看着它在对话里一段段做完"。后台定时任务才保留旧的看板+回执。

## 7. 数据形态（落地锚点）

> 都建在现有结构上，不另起炉灶。

- **task span 标记**：`task.start(goal, success_criteria, parent_session_id)`；在 `AITaskJob` 增 `is_foreground`、
  `span_start_msg_id` / `span_end_msg_id`（前台任务复用当前 session，不新建 `session_task_*`）。
- **阶段目标**：一段 span 内可有多个有序 `goal`，每个含 `{goal, success_criteria, status, tool_trace_ref}`。
- **轨迹（trace）**：`task.complete` 时从本 span 的 `mcp_tool_call` 气泡聚合：
  `[{step, goal_id, tool, args, result_digest, ok, error, duration_ms}]`。这正是 `进化流程设计.md` 阶段二要的"结构化原料"。
- **沉淀载体**：沿用 `SKILL.md`，在 frontmatter 增 `maturity`(draft/soft/hard)、`success_count`、`last_used_at`、`params`(变量声明)。
- **检索**：任务开场复用 `librarian.brief()`（触发词），命中则把候选 `SKILL.md` 注入，由 AI 决定是否采用。

## 8. 落地路线（每步独立可交付）

| 步 | 内容 | 交付标准 |
| --- | --- | --- |
| 1 | 前台 `task.start/complete` 走对话内 span，不新建 session；对话内显示"任务中+阶段目标"内联条 | 用户在对话里能开/关一个任务，无独立会话、无信件 |
| 2 | `task.complete` 时结构化抽取轨迹并 propose 沉淀为 `draft` 级 `SKILL.md` | 一次任务后能看到含"目标→工具→参数"的草稿流程 |
| 3 | 任务开场用 `brief()` 检索候选流程，AI 读 `soft` 级 `SKILL.md` 重放；出错自动校正并回写、`success_count`++ | 同类任务第二次明显更顺，且流程被更新 |
| 4 | 成熟度门槛：连续成功 + 判据稳定 → 升 `hard`；串联多 MCP、出错才接管；成功率下降自动降级 | 高频稳定流程能"一口气跑"，异常时回到人/LLM |
| 5 | UI 瘦身：砍前台新建表单、任务栏目收敛为只读看板+定时配置 | 任务相关界面明显变简单 |

> 步骤 2-4 的"沉淀去重/审批门禁/安全回归/度量淘汰"全部复用 `进化流程设计.md`，本文不重复实现。

## 9. 开放问题（需你拍板）

1. **进入任务模式的触发**：纯关键词匹配（简单但易误触发） vs 意图识别+轻确认（稳，多一步交互）。**建议后者。**
2. **毕业门槛 N**：软→硬需要"连续成功几次"？建议从 N=3 起步，按度量调。
3. **硬复用边界**：哪些类别的流程**永不**升 hard（如涉及写操作/提权/外部副作用的）？建议与 `进化流程设计.md` P2 风险分级对齐，高风险一律留在 soft + 人审。
4. **前台 vs 后台的代码边界**：是在 `AITaskJob` 上加 `is_foreground` 分叉，还是前台任务彻底不落 `AITaskJob`、仅作为会话内标记？影响 §7 数据形态，需先定。
