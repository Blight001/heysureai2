# 执行摘要与跨章依赖图（先读这页）

## 一句话定位

HeySure AI 要从「逐动作云端遥控的 Agent」升级为「**有常驻记忆、可沉淀技能、能安全分发能力、由策展者传承知识**的数字社会」。
核心方法论贯穿全文：**地基项目里大多已有，缺的是把它们连起来的「管道」，最大化复用、不新增骨架。**

## 三套"技能/能力"抽象的关系（实现前必须先对齐）

全文出现三个名字相近、容易各做各的概念，**它们不是竞品，而是不同语境下的同一谱系**：

```
                          ┌──────────────────────────────────────┐
                          │  Skill（第二章 §2 的统一抽象）          │
                          │  type = procedure | instruction        │
                          └──────────────┬───────────────┬────────┘
              procedure（动作重放，端侧执行，不过 LLM）   instruction（方法论知识，命中 trigger 注入上下文）
                          │                               │
                          ▼                               ▼
        ┌──────────────────────────┐      ┌──────────────────────────────────┐
        │ skill_card（第三章）        │      │ 复用 KnowledgeEntry（第二章 §2.3）  │
        │ = Skill 的 procedure 形态   │      │ = Skill 的 instruction 形态        │
        │ 服务端存储 / 端侧 card.execute│      │ trigger 命中 → 注入正文            │
        └──────────────────────────┘      └──────────────────────────────────┘

        ┌──────────────────────────────────────────────────────────────────┐
        │ Capability（第五章 §7.4）—— 另一个语境：Claude Code 端的"能力包"      │
        │ kind = skill（SKILL.md） | tool（MCP tool 包）                      │
        │ 关注的是「装载到哪个 AI / AI 间转授 / 来源信任分级」，而非重放或注入    │
        └──────────────────────────────────────────────────────────────────┘
```

- **第二章 Skill** 是服务端 Agent 的技能抽象，把第三章 `skill_card`（procedure）与指令式知识（instruction）收口为一个概念。
- **第三章 skill_card** = 第二章 Skill 的 `procedure` 形态的具体实现，根本不进上下文、端侧确定性重放。
- **第五章 Capability** 处在 **Claude Code 端侧**这一不同语境：它打包的是 Claude 的 `SKILL.md` / MCP tool，关心的是分发、转授、来源信任，与前两者**不是同一条执行链**。
- 待办：实现时需一处统一命名/映射，避免 Skill / skill_card / Capability 三套数据结构各自漂移。

# 第二章 记忆与技能体系-设计方案

# 记忆与技能体系 - 设计方案

本文设计三件相互关联的能力，对齐 Codex `AGENTS.md` / Claude Code `CLAUDE.md` 这类
Agent 框架的核心特性，并落到 HeySure AI 2.0 现有架构上：

1. **项目记忆文件**：人可编辑、每轮常驻注入、分层覆盖的上下文文件（最大缺口）
2. **Skill 双形态**：把现有 `skill_card`（确定性重放）与新增的「指令式 Skill」统一
3. **长期记忆**：把已有的 `Memory` / `KnowledgeEntry` / `Valhalla` 补成「写入→巩固→检索→注入→遗忘」闭环

核心判断：**这三件事的地基项目里都已经有了，缺的是把它们连起来的「管道」**，而不是新骨架。
设计原则是最大化复用现有 section 注入机制、librarian 文件存储模式与信任分/衰减字段。

## 0. 现状盘点：不要重造的部分

| 诉求 | 已有对应物 | 文件锚点 | 缺口 |
| --- | --- | --- | --- |
| 项目记忆文件 | `Memory` 表（project_id 维度检索式记忆） | `server/api/models/knowledge.py` / `tools/memory.py` | 缺「常驻注入、人可编辑的文件层」 |
| Skill | `skill_card`（录制→重放→自愈→晋升，设计已到 S6） | `doc/沉淀技能卡片-设计方案.md` | 只覆盖「程序性重放」一类，缺「指令式知识」 |
| 长期记忆 | `Memory` + `KnowledgeEntry` + `Valhalla` + `EvolutionInput` + `Archivist` | `models/knowledge.py` / `services/librarian_service.py` | 检索是子串匹配、巩固/自动抽取/衰减未成闭环 |

### 0.1 一个已经无意中成立的记忆三分法

项目现有的数据结构恰好对应认知科学的记忆分类，**这是资产，后续设计沿用、不打散**：

| 记忆类型 | 系统中的载体 | 回答的问题 |
| --- | --- | --- |
| 情景记忆 episodic | `Valhalla` + 聊天记录 | 「发生过什么」 |
| 语义记忆 semantic | `Memory` 表（fact / decision / lesson） | 「关于世界的事实」 |
| 程序记忆 procedural | `KnowledgeEntry`（怎么做） + `skill_card`（确定性重放） | 「怎么做某事」 |

### 0.2 已有的注入管道（直接复用）

Prompt 不是一整块拼死的，而是**按 section 组装**，这正是项目记忆文件要挂靠的地方：

- `server/api/chat_runtime/chat_prompt_utils.py`：`_append_prompt_section(text, title, body)` /
  `_strip_prompt_section(text, title)` —— 按标题增删 section。
- `server/api/chat_runtime/run_state.py`：`_AUTO_RUNTIME_SECTION_TITLES` —— 运行时注入的
  section 标题注册表（如 `AI 工作目录`），重组前会先 strip 再 append，避免重复堆叠。
- `server/api/chat_runtime/chat_runtime_helpers.py:50`：现成示例——
  `_append_prompt_section(system_prompt, "AI 工作目录", get_project_root(user.id, cfg.id))`。
- `server/api/chat_runtime/chat_scheduler.py:158`：librarian 的 **渐进披露注入**——`brief()`
  把「知识标题 + 触发词 + 摘要」塞进 prompt，全文按需 `librarian.read(memory_id)` 才拉。

> 结论：**「每轮常驻注入一段文本」的能力已经存在**。项目记忆文件 = 新增一个常驻 section +
> 一个文件存取层，复用度极高。

## 1. 项目记忆文件（AGENTS.md / CLAUDE.md 对标）

### 1.1 定位：和 Memory 表互补，不是替代

`Memory` 表回答「**搜得到**」（要 `memory.search` 才出来）；项目记忆文件回答「**忘不掉**」
（每轮无条件在场）。两者解决不同问题，不合并：

| | Memory 表 | 项目记忆文件 |
| --- | --- | --- |
| 进入上下文的方式 | 检索命中才注入 | 每轮常驻注入（受 token 预算约束） |
| 编辑者 | AI 写、用户少改 | 用户为主、AI 可 append |
| 内容性质 | 离散事实/教训 | 项目约定、目标、术语、禁忌（稳定、需常看） |
| 体量 | 多、可无限增长 | 少而精，有硬上限 |

### 1.2 分层覆盖：细的盖粗的

参考 Codex/Claude 的层级合并语义，由粗到细三层，**下层覆盖上层同名条目**：

```
global   用户/组织级：通用语气、全局禁忌、跨项目规范
project  项目级：目标、技术栈、术语表、协作约定、目录说明
agent    AI 成员级：该成员的角色补充、个人偏好
```

注入顺序 `global → project → agent`，越靠后优先级越高（和 prompt 末尾覆盖前文的语义一致）。

### 1.3 存储：复用 librarian 的「真相在文件 / DB 做索引」模式

完全沿用 `librarian_service` 既有约定（`<workspace_root>/KnowledgeBase/...`），新增一个并列目录：

```
<workspace_root>/Memory/
  global.md                      # global 层，单文件
  agents/<ai_config_id>.md       # agent 层，每成员一文件
  projects/<project_id>/AGENTS.md# project 层（命名 AGENTS.md，对标 Codex，降低用户心智成本）
  index.json                     # 三层文件清单 + 摘要 + 字节数（前端浏览/预算用）
```

> `workspace_root` 取法见 `server/api/core/config.py` 的 `user_workspace_dir(user_id)`；
> project 根目录取法见 `get_project_root(user.id, cfg.id)`（`chat_runtime_helpers.py` 已在用）。

DB 侧**不新建大表**，复用 `KnowledgeEntry` 即可（它已有 `scope`=`global/ai:<id>/project:<id>`、
`file_path`、`summary`、`status`），只加一个语义位区分「常驻记忆文件」与「按需知识」：

```jsonc
// KnowledgeEntry 增加一列
"entry_kind": "memory_file" | "knowledge"   // memory_file=常驻注入；knowledge=按需 read（默认）
```

这样项目记忆文件天然进 `KnowledgeBase/index.json` 的浏览/检索体系，前端零改造即可列出。

### 1.4 注入：新增一个常驻 section（核心改动，约 30 行）

在 system prompt 组装处，于 `AI 工作目录` section 之后追加 `项目记忆` section：

```python
# server/api/chat_runtime/chat_runtime_helpers.py 组装链路中
mem_text = build_project_memory(user.id, cfg.id, project_id, budget_tokens=1200)
if mem_text:
    system_prompt = _append_prompt_section(system_prompt, "项目记忆", mem_text)
```

配套两步（与既有机制一致）：

1. **注册 section 标题**：把 `"项目记忆"` 加入 `run_state.py` 的 `_AUTO_RUNTIME_SECTION_TITLES`，
   保证重组 prompt 时先 strip 旧的再 append 新的，不堆叠。
2. **token 预算**：`build_project_memory` 按 `global → project → agent` 顺序填充，累计字节
   超 `budget_tokens` 时，对超限层**降级为摘要 + 提示「全文用 `project_memory.read` 拉取」**
   （直接套用 librarian `brief()` 已有的渐进披露策略）。

### 1.5 读写接口：新增 MCP 工具 `project_memory.*`

```
project_memory.read(scope, scope_target?)        读某层全文（注入被预算截断时用）
project_memory.append(scope, scope_target?, text) AI 追加一条（带时间戳，避免覆盖人写的内容）
project_memory.replace_section(scope, heading, body)  按二级标题替换（结构化更新）
```

- AI 默认只能 `append`，`replace_section` 需更高权限（防止 AI 把用户写的约定改没）。
- `global` 层的写入建议仅 `Archivist` 或用户可做（影响所有成员，风险高）。
- 落点：`server/mcp_runtime/mcp/tools/project_memory.py`，注册进 `registry.py`，权限规则进
  `permissions.py`（参考现有 librarian 的 `_ensure_librarian_only` 收敛写权限的做法）。

### 1.6 前端：复用「沉淀技能/知识」页式样

`web/` 加「项目记忆」编辑页：三层 tab、markdown 编辑器、保存即写文件 + 刷新 `index.json`。
和技能卡片设计文档 S5 规划的「沉淀技能」管理页同构，组件可复用。

## 2. Skill 双形态：统一 procedure 与 instruction

### 2.1 你只做了一半的 skill

现有 `skill_card` 是**确定性动作重放**（RPA 宏的治理化升级），擅长桌面/浏览器的连贯机械操作。
但 Codex/Claude 语义里的 Skill 还有另一种形态，项目目前没有：

| | `skill_card`（已有，procedure） | 指令式 Skill（建议补，instruction） |
| --- | --- | --- |
| 本质 | 固化的**动作序列**，端侧重放不过 LLM | 一段 **markdown 方法论/指令** + 可选脚本，按需加载进上下文 |
| 适合 | 「点保存→输文件名→回车」机械操作 | 「怎么写周报」「审代码看哪几点」需 LLM 现场发挥的知识 |
| 加载方式 | `card.execute` 端侧执行 | 命中 trigger 时把正文**注入上下文**（渐进披露） |
| 失败模式 | 锚点失效 → 视觉自愈（§沉淀技能 §4.3） | 指令不适用 → AI 自行判断偏离 |

### 2.2 统一抽象：一个 Skill，两种 type

不新建并行体系，把两者收进同一概念：

```jsonc
{
  "skill_id": "...",
  "name": "...",
  "type": "procedure" | "instruction",  // procedure → 现有 skill_card；instruction → 新增
  "triggers": ["写周报", "周报"],         // 复用 KnowledgeEntry.triggers
  "scope": "private | team | public",
  "body_ref": "...",                      // instruction: markdown 路径；procedure: 卡片 steps
  "status": "draft | supervised | trusted | deprecated"
}
```

### 2.3 指令式 Skill 几乎是白嫖

`KnowledgeEntry` 已经具备指令式 Skill 所需的几乎全部字段：

| 指令式 Skill 需要 | KnowledgeEntry 已有 | 文件锚点 |
| --- | --- | --- |
| 触发关键词 | `triggers` | `models/knowledge.py:53` |
| 作用域 | `scope` / `scope_target` | `models/knowledge.py:54` |
| markdown 正文 | `file_path`（topics/<slug>.md） | `librarian_service.py:9` |
| 渐进披露注入 | `brief()` 注标题+触发词，`read` 拉全文 | `chat_scheduler.py:158` |
| 使用统计/晋升 | `use_count` / `status` / `confidence` | `models/knowledge.py:60` |

**唯一要补的逻辑**：trigger 命中后，从「列标题」升级到「自动注入正文」（或 AI 主动 `read`）。
即在 `brief()` 注入的基础上，对 `entry_kind=skill_instruction` 且 trigger 强命中的条目，
直接把正文拼进一个 `可用技能` section（仍受 token 预算约束）。

### 2.4 与项目记忆文件的分工

- **项目记忆文件**：无条件常驻（项目级稳定约定）。
- **指令式 Skill**：trigger 命中才加载（特定任务才用的方法论），避免塞爆上下文。
- **procedure skill_card**：根本不进上下文，端侧执行。

三者构成「常驻 / 按需 / 端侧」三档加载策略，token 效率最优。

## 3. 长期记忆：补齐五段管道

长期记忆系统的标准链路 **写入 → 巩固 → 检索 → 注入 → 遗忘**。项目骨架齐全，弱在「检索」和
「巩固」，且「写入」尚未自动化。

```
写入(extract) ─▶ 巩固(consolidate) ─▶ 检索(retrieve) ─▶ 注入(inject) ─▶ 遗忘(decay)
   ▲ 自动抽取          ▲ 去重/合并/         ▲ 子串→向量        ▲ 预算管理       ▲ 评分归档
     未做                矛盾标记弱           最该升级            已有 section     字段已备
```

### 3.1 写入（extraction）：让 Archivist 顺手记

现状 `memory.write`（`tools/memory.py:75`）是 AI 手动调，依赖模型自觉。改进：

- **会话/任务结束**触发自动抽取——`Archivist` 已在做总结（见 `数字社会核心管理员` 文档），
  在总结产物里顺手提「值得长期记的事实/教训/决策」落 `Memory`，`source` 记来源 message/job。
- 抽取走「候选 → 入 `pending` → 巩固时定夺」，不直接污染主记忆（与 librarian propose 同构）。

### 3.2 检索（retrieval）⚠️ 最该升级：子串 → 向量

现状 `_memory_search` 是 `query in haystack` 的**子串匹配**（`tools/memory.py:138`），
语义近似（「保存文件」vs「存盘」）搜不出来。

- 项目用 Postgres → **上 `pgvector`**。给 `Memory` / `KnowledgeEntry` 加 `embedding` 列。
- 检索改为 **向量召回 Top-K + 关键词/标签兜底**，再按 §3.5 的衰减分重排。
- SQLite 回退场景（README 提到无 DB 时回退 SQLite）：保留现有子串匹配作为 fallback，
  按 `DATABASE_URL` 是否 Postgres 选择检索后端。

### 3.3 巩固（consolidation）：去重 / 合并 / 矛盾

库一大就会冒出 50 条「差不多的记忆」——这与技能卡片设计文档 **§6.5 去重** 是同一个问题，
解法对称：

1. **语义去重**：写入时对相似记忆做向量近邻，超阈值 → 提示合并而非新建。
2. **矛盾检测**：新记忆与既有高 `confidence` 记忆冲突 → 标 `conflict`，交 `Archivist` 裁决。
3. **周期巩固**：`Archivist` 定期把同主题碎片记忆压成一条高置信记忆（情景 → 语义的提升）。

### 3.4 注入（injection）：与项目记忆共用预算

检索结果按 token 预算拼进 prompt，复用 §1.4 的 section 机制，新增 `相关记忆` section。
和 `项目记忆`、`可用技能` 共享同一份预算池，按优先级 `项目记忆 > 命中技能 > 检索记忆` 分配。

### 3.5 遗忘（decay）：字段都在，只差公式

`Memory` / `KnowledgeEntry` 已有 `confidence` / `use_count` / `last_used_at` / `archived`
（`models/knowledge.py:19,60-61`），只需补一个评分与定时归档：

```
score = w1 * recency(last_used_at)      # 越近越高
      + w2 * frequency(use_count)        # 用得越多越高
      + w3 * confidence                  # 置信越高越高
低于阈值且长期未用 → 自动 archived（不删，可恢复），不进检索/注入
```

`Archivist` 周期任务跑评分，与 §3.3 巩固合并为一个「记忆维护」作业。

# 第三章 沉淀技能卡片-设计方案

# 沉淀技能卡片 - 设计方案

本文记录"沉淀技能卡片"（Skill Card）功能的完整设计。目标是回答四个问题：

1. 为什么要做技能卡片，它在系统中是什么定位
2. 卡片的数据模型怎么设计才能跨 AI、跨环境共享
3. "录制 → 重放 → 失败自愈 → 晋升"的完整生命周期怎么跑
4. 共享带来的新坑（环境可移植性、权限、信任、并发、发现、安全）怎么逐个解决

## 1. 背景与定位

### 1.1 要解决的核心问题

当前桌面/浏览器操作是"逐动作云端遥控"：云端 LLM 每生成一个工具调用，就经过
`dispatch → socket → 轮询` 一个往返才执行下一步（见 `server/ai_runtime/inference/core.py`
的 `_run_worker_impl` 主循环与 `_dispatch_endpoint_via_runtime` 的轮询）。一段
10 步的连贯操作会被拆成 10 次 LLM 推理 + 10 次往返，延迟高、成本高、且每步都可能漂移。

技能卡片把**一段已验证成功的动作序列固化下来**，下次直接本地重放，不再逐步过 LLM：

- **延迟**：重放在 endpoint（桌面/浏览器）本地一口气跑完，不逐步往返云端
- **成本**：常规操作零 token
- **稳定**：确定性重放比每次重新推理更可靠
- **出错可自愈**：重放失败时停在失败步，把现场回传 AI 检查并修改卡片

### 1.2 系统定位

技能卡片是 README 中 `KnowledgeBase` / `skill` 演进方向的具体落地，是可被沉淀、
检索、共享、传承的**数字资产**，而不是一次性宏录制。它遵循项目四原则：

| 原则 | 在卡片上的体现 |
| --- | --- |
| 可治理 | 卡片有 surface 绑定、capability 契约、信任分级、共享范围 |
| 可追溯 | 每张卡有版本历史、执行统计、失败现场记录 |
| 可组合 | 卡片可参数化、可被 composite 卡编排、跨端协作 |
| 可传承 | 存数据库、跨 AI 共享，`Archivist` 负责策展与审核 |

### 1.3 关键决策：服务端存储 + 跨 AI 共享

卡片**不存本地**，而是作为服务端网页的"沉淀技能"保存到数据库，后续其他 AI 可直接
调用。这把卡片从"本地宏"升级为"共享技能库"，同时引入一批本地方案不存在的新坑
（见第 6 章），所有坑的根因都是同一句话：

> "共享"把"在我机器能跑"变成了"在任意机器、任意 AI、任意权限下都得安全可控"。

## 2. 核心概念：分类不是单一标签，是正交维度

只按 "Windows 类 / 浏览器类" 分不够用。卡片有四个互相独立的维度，全部进 schema：

```
surface     (执行端，硬绑定)   : windows | browser | shell | composite
capability  (能力契约，硬校验): 卡片会调用哪些工具 → 决定谁能跑
domain      (业务域，软检索)   : 文件操作 | 数据录入 | 网页抓取 | 表单填写 ...
scope       (共享范围)         : private(本AI) | team | public
```

- **surface 是硬绑定**：Windows 卡不可能在浏览器跑，调用时先按它过滤。对应
  `server/connector_runtime/dispatch/desktop_agent_tools.py` 里 `endpoint_tools_for_config`
  已有的 endpoint-agent 类型。
- **capability 是硬校验**：卡片声明它要用 `shell.run / fs.write`，运行时拿它与
  调用方的 `effective_tool_allowlist` 取交集（见第 6.2 节）。
- **domain 是软检索**：用于语义搜索、去重，不参与权限。
- **scope 是共享范围**：决定卡片对谁可见、谁能改（见第 6.4、6.6 节）。

不同 surface 的定位锚点不同，但 schema 共享、由 `surface` 字段选择定位策略：

| surface | 定位锚点优先级 |
| --- | --- |
| windows | UI Automation 控件名 > 文本/控件类型 > 图像模板 > 坐标(兜底) |
| browser | 稳定 DOM selector / aria-label > 文本 > XPath > URL 模式 |
| shell | 无定位，按命令 + 期望输出断言 |
| composite | 自身无锚点，编排子卡片，按子卡片各自 surface 定位 |

> 视觉识别（截图认图标）不进锚点表，它属于**发现层**，见第 2.1 节。锚点表里的都是
> **可确定性重放**的策略（模板匹配 / 控件查询都不过 LLM）；视觉只在冷启动和兜底时介入。

### 2.1 视觉是发现层，锚点是固化层

一个核心矛盾：卡片要的是**确定性重放**（不过 LLM），但"AI 不认识某个图标、得截图看或靠
鼠标指一下"本质是**非确定性的视觉识别**（每次都得过模型的眼睛）。这两件事直接打架——
若把"截图→让 AI 判断点哪"录进卡片，重放还是每步过 LLM，等于没沉淀。

解法是把"找"和"用"拆成两个阶段：**首次靠视觉找，命中那一刻把"找到的结果"固化成锚点；
重放时直接用锚点，全部失败才退回视觉。**

```
首次（慢，AI 在场）：截图 → AI 判断 → 点击
   命中后系统抄下的不是"截图+判断"过程，而是命中点能提取的稳定特征：
     UIA 控件名/类型(最稳) > 周围图像模板(次稳) > 相对锚点 > 坐标(兜底)
重放（快，无 LLM）：按锚点优先级定位命中即用；全部失败 → 该步退回 AI 重新视觉发现
```

两种"找图标"的方法各自的归位：

- **截图法**：是锚点提取的**输入**，不是卡片内容。AI 截图找到图标后，系统从命中点抓
  控件特征 + 周围图像块存进 step；重放用**模板匹配**（确定性）去屏幕找它，而不是再让 AI 看一次。
- **鼠标光标指示法**：用光标指一下目标，等于给锚点提取喂一个**人类标注的精确点**，质量比
  AI 截图猜更高。建议做成录制里的一等动作："把鼠标移到目标上，我来记住它的特征"。

结论：**视觉永远跑不掉，但从"每次都做"降级成"首次 + 兜底时才做"** —— 这正是卡片省时省钱的来源。

### 2.2 作用域锁定：先确定在哪个窗口找，再谈怎么找

这是消歧的**第一层前置门**，不是 §2.3 的并列选项。例子：QQ 和微信里都有"张三"头像，
甚至长得一样。在 QQ 里识别张三，绝不能去微信窗口找——否则再强的相对锚点也没用，因为
微信里"张三那一行"同样成立，相对锚点自己也会撞车。

**所以作用域锁定必须发生在任何识别之前。先把战场锁死在目标窗口，再在窗口内消歧。**
作用域是一条由粗到细的链，进程/窗口层是**硬隔离**：

```
scope_chain:  process(QQ.exe) → window("张三") → panel(聊天列表) → row(唯一文本)
              └─ 进程级硬隔离（本节）──┘ └────── §2.3 容器内消歧 ──────┘
```

实现上，进程/窗口层做两件事：

1. **解析目标窗口**：按进程名 / 标题找到目标窗口（句柄 + 边界框）。
2. **物理裁剪识别范围**：识别用的截图只截该窗口客户区，别的应用像素**根本不进入搜索画面**，
   从源头杜绝跨应用串台。

卡片头部因此要声明**应用作用域**，且参数化（窗口标题会随联系人变，不能硬绑）：

```jsonc
"app_scope": {
  "process": "QQ.exe",                 // 进程名，硬隔离的根
  "window_match": "{{contact_name}}",  // 窗口/标题参数化
  "on_missing": "halt"                 // 目标应用没开 → 停手，绝不去别的窗口找
}
```

`on_missing: halt` 是 QQ/微信例子的安全底线：**目标窗口不存在时必须停，绝不退而求其次。**

> 能力依赖（见 §5.1）：进程过滤现成可用——`window.list` 已回传 `{hwnd, pid, title, name(进程名)}`，
> 能按 `name` 过滤出 QQ 窗口、排除微信。但**按窗口物理裁剪截图暂缺**：`window.list` 还没回传
> 窗口边界框（x/y/w/h），无法喂给 `screen.capture_region`。第一版只能做到"进程过滤 +
> `window.focus` 把目标窗口置顶后再全屏识别"（弱隔离）；真正的像素裁剪需桌面端给 `window.list`
> 补回传窗口 rect 后才能上（列入 §5.1 / S4.5）。

### 2.3 多目标消歧：同一窗口内出现多个相同图标

作用域锁定后，若**同一窗口内**仍有多个一模一样的目标（聊天列表每行都有头像、列表每行都有
删除图标），盲点第一个就点错行。这是 RPA 最经典的翻车点，尤其撞上 destructive 步骤代价很大。
三条铁律：

**铁律一：目标 = 图标 + 唯一性约束，而不是图标本身。** 单凭图标永远可能撞车，让它唯一
靠的是**它周围的唯一信息**。三种消歧手段，按优先级：

1. **区域收窄（region）**：先把搜索范围缩到一个子区域，候选自然塌缩成 1。
   "在**当前弹出的对话框**里找确定按钮"，而不是全屏找。很多重复只是因为搜索范围太大。
2. **相对锚点（anchor + relation）—— 最关键**：不直接定位图标，先定位一个**唯一的邻居**，
   再相对它找图标。"在那一行文字是 `报告.docx` 的行里点删除图标"——行文本唯一，图标跟着唯一。
   把唯一性从长得一样的图标，转移到它所在行/容器的唯一文本上。
3. **序号（ordinal）—— 最次，仅兜底**："第 3 个匹配"。列表一变就错，只在前两种都拿不到时用。

**铁律二：数不清就停手，绝不乱点。** 每个点击步带护栏 `expect_count`（默认 1）。重放定位后
先数命中几个：

```
恰好 1 个        → 点
0 个             → 锚点失效，走 vision_fallback / 自愈（§2.1）
≥2 个且无法消歧  → 【停！不点】halt，回传"歧义：N 个候选 + 截图"，退回 AI 判断（§4.3）
```

宁可停下问 AI，也不能猜一个点下去——destructive 步骤尤其如此。复用已有的
`on_fail: halt` + 失败现场回传，零新增骨架。

**铁律三：歧义在录制时就检测，不留到重放才炸。** 录制的锚点提取阶段，抽到锚点后当场在
屏幕上数它匹配几个：匹配 1 个 → 干净锚点直接存；匹配多个 → 必须补一个消歧器（把实际点中的
那一个的唯一行文本 / 最近唯一标签 / 序号记下来）。歧义在录制时被固化成消歧规则，而非埋雷。

## 3. 数据模型

卡片 schema 一开始就要把可移植性、权限、参数、前置条件、版本、共享范围设计进去；
否则先做成扁平动作列表再加共享会非常痛。

### 3.1 卡片本体（card）

```jsonc
{
  "id": "card_a1b2c3",
  "name": "保存当前文件并命名",
  "description": "在编辑器中触发保存对话框，输入文件名并保存",
  "surface": "windows",
  "domain": ["文件操作"],
  "scope": "team",
  "owner_ai_config_id": 12,          // 创建者
  "version": 3,                      // 当前版本号
  "status": "trusted",              // draft | supervised | trusted | deprecated
  "record_mode": "teach",          // auto(AI自动抄录) | teach(用户逐步教学)，见 §4.0
  "capability": ["keyboard.type", "mouse.click"],   // 需要的工具（权限契约）
  "app_scope": {                       // 作用域锁定：识别只在此窗口内进行（§2.2）
    "process": "QQ.exe",
    "window_match": "{{contact_name}}",
    "on_missing": "halt"
  },
  "params": [
    { "name": "filename", "type": "string", "required": true, "secret": false }
  ],
  "preconditions": [
    { "check": "active_window_matches", "value": "编辑器" }
  ],
  "steps": [ /* 见 3.2 */ ],
  "postconditions": [
    { "check": "window_closed", "value": "保存对话框" }
  ],
  "environment_signature": "win11|1920x1080|...",  // 录制环境，用于信任分归属
  "created_at": "...",
  "updated_at": "..."
}
```

### 3.2 步骤（step）—— 每步必须带断言

"出错返回 AI" 要有用，前提是知道**错在第几步、期望 vs 实际是什么**。所以每个 step
带 postcondition（assert）。执行器跑完一步若断言不过，就停在该步并打包现场回传。

每个 step 的 `target` 不是单一策略，而是**一串带优先级的锚点 + 一个 vision 兜底**（见
第 2.1 节）。重放时锚点逐个试，命中即用（零 LLM）；全部失败才触发 `vision_fallback`，
该步退回 AI 重新视觉发现，并用新命中点回写锚点（即失败自愈，卡片越用越准）。

```jsonc
{
  "index": 3,
  "act": "click",
  "target": {
    "anchors": [                              // 按优先级排列，确定性重放，不过 LLM
      { "strategy": "uia",   "name": "保存", "controlType": "Button" },
      { "strategy": "image", "ref": "blob://save_icon.png", "threshold": 0.9 },
      { "strategy": "coord", "x": 1024, "y": 768 }   // 最后兜底
    ],
    "vision_fallback": {                      // 锚点全失败才触发：退回 AI 视觉发现
      "enabled": true,
      "hint": "工具栏上软盘形状的保存图标"
    }
  },
  "args": {},
  "assert": {                                 // 执行后的期望状态
    "check": "window_closed",
    "value": "保存对话框",
    "timeout_ms": 3000
  },
  "on_fail": "halt",                          // halt | retry(n) | skip
  "destructive": false
}
```

> 锚点能力分阶段落地（见 §5.1 桌面端现状）：第一版受桌面工具回传限制，`anchors` 可能
> 只有 `image` + `coord`；`uia` 控件锚点需桌面 endpoint 在点击命中时补回传控件信息后才能用。

多目标消歧示例（见 §2.3）—— 列表里每行都有删除图标，靠唯一行文本定位 + 护栏：

```jsonc
{
  "index": 4,
  "act": "click",
  "target": {
    "region": { "scope": "window", "value": "下载列表" },   // ① 先把搜索范围收窄
    "disambiguate": {                                        // ② 相对唯一邻居定位
      "anchor": { "strategy": "text", "value": "{{filename}}" },  // 行内唯一文本
      "relation": "same_row",                               // same_row | right_of | below | offset
      "then": { "strategy": "image", "ref": "blob://delete_icon.png" }
    },
    "ordinal": null,                                         // ③ 仅前两者都拿不到时用"第 N 个"
    "expect_count": 1,                                       // 护栏：命中数≠1 不点
    "vision_fallback": { "enabled": true, "hint": "该文件那一行的删除按钮" }
  },
  "args": {},
  "assert": { "check": "row_removed", "value": "{{filename}}", "timeout_ms": 3000 },
  "on_fail": "halt",
  "destructive": true                                        // 删除类，命中歧义必须停手
}
```

参数化示例（避免为每个文件名存一张卡）：

```jsonc
{ "index": 2, "act": "type", "args": { "text": "{{filename}}" } }
```

### 3.3 版本历史（card_version）

共享卡 + 自愈编辑 = 多 AI 可能改同一张卡，必须保留版本历史与作者，支持回滚与 fork

```jsonc
{
  "card_id": "card_a1b2c3",
  "version": 3,

  "change_summary": "第3步保存按钮锚点改为 aria-label 兜底",
  "snapshot": { /* 该版本完整卡片 */ },
  "created_at": "..."
}
```

### 3.4 执行统计（card_run_stat）—— 按环境维度

```jsonc
{
  "card_id": "card_a1b2c3",
  "environment_signature": "win11|1920x1080",
  "version": 3,

```

## 4. 生命周期

```
首次：AI 逐步操作（慢路径）──成功──▶ 录制为 draft 卡片
之后：card.execute 本地快速重放
出错：停在失败步 → 回传现场(failed_step + 期望/实际 + 截图) → AI 改卡 → version++ → 从失败步续跑
稳定：在某环境连续成功 N 次 → 该环境下升级 supervised → trusted → 完全自动
跨环境：被新环境/新 AI 调用时，信任分按环境重新计，回落 supervised
```

### 4.0 两种录制模式：auto 抄录 vs teach 教学

录制不是只有一种。按"谁主导、断言/消歧从哪来"分成两种模式，进 schema 的 `record_mode`
字段（§3.1）：

```
auto   AI 自挂开关、流经的动作自动抄录（§4.1）。快，但断言、消歧器、脱敏都得 AI 事后猜。
teach  用户逐步指挥 + 逐步确认。慢，但断言、消歧、脱敏全部由人当场钉死，质量高。
```

两种模式**共用同一套底层**——都走 `core.py` dispatch 咽喉点抄录（§4.1）、都在 `recorder.stop`
做锚点提取 / 脱敏 / 噪声过滤、都存成 draft。teach 只是在循环里多了"人下指令 + 人确认"两步，
**不新增抓取骨架**。

**关键澄清：teach 不是"用户亲手操作、AI 在旁偷看"。** 那需要 hook 操作系统裸输入事件，与 §4.1
明确拒绝的方向冲突。teach 的正确形态是 **用户用自然语言逐步指挥、AI 执行并记录**——AI 仍是
那双手，人只负责"教 + 确认"。这样仍复用咽喉点，零新增骨架。

teach 的录制循环：

```
recorder.start(mode="teach", name, surface, ...)
循环：
  用户自然语言指令（"点工具栏上的保存按钮"）
    → AI 执行对应动作（mouse.click / keyboard.type / ...）
    → AI 当场回问三件事：
        ① 这一步对吗？             —— 确认动作正确，错了立刻重做，不留错步
        ② 之后该出现什么？         —— 人确认的断言，质量远高于自动生成（治 §4.1-3 痛点）
        ③ 命中多个时：你指哪个？   —— 当场钉死消歧器，落实 §2.3 铁律三（歧义不留到重放）
    → 抄录 { 动作 + 人确认的 assert + 人确认的消歧器 }
  （敏感输入时用户可直接说"这个存成参数、别记" → 比自动猜更可靠的脱敏，见 §6.6）
recorder.stop()  → 加工存成 draft
card.execute(dry_run=true)  → 用户看 AI 完整跑一遍"查看学习效果"
    → 对   → 允许直接晋升 supervised（teach 卡质量已被人确认过）
    → 错   → 进 §4.3 自愈：回传失败现场 → 用户/AI 改那一步 → version++ → 从失败步续跑
```

后半段"后续自行运作、出错了再和用户沟通完善"就是现成的 §4.3 自愈循环，只是失败现场回传
**多一条人工通道**（不止回给 AI，也能回给用户），基本不动骨架。

**两种模式怎么选（不是默认全用 teach，教学费人力）：**

| 卡片类型 | 推荐模式 |
| --- | --- |
| 简单、低风险、仅本机自用 | `auto` 快速抄录 |
| 重要 / 含 destructive 步骤 / 要跨环境跨 AI 共享 / 要进 public | `teach` 人教，质量够格才允许升 scope |

teach 模式同时缓解了本方案两个最脆的环节：**断言生成**（§4.1-3，由猜变问）和**录制脱敏**
（§6.6，由 AI 猜敏感值变人工标注）。

### 4.1 录制（draft）—— AI 自挂咽喉点

**录制思路：AI 自己开启录制开关，之后所有动作被自动抄录，停止时加工成卡片。**

不去 hook 操作系统输入事件，而是复用系统天然的咽喉点：所有桌面/浏览器操作都必经
`server/ai_runtime/inference/core.py` 的 `_run_worker_impl` 循环 → `_call_mcp_or_endpoint_tool`
这一处 dispatch。录制 = 在这个点挂一个开关，流经的 endpoint 操作连同参数和结果顺手抄一份。
好处：AI 的每个动作本来就是一次**结构化工具调用**，参数齐全、语义清晰，无需抓裸输入事件。

落地形态 —— 三个新 MCP 工具 + 一段拦截逻辑：

```
recorder.start(name, surface, mode, ...) 在 run session context 里置 recording 标志 + 开 buffer（mode=auto|teach，见 §4.0）
（AI 正常执行 mouse.click / keyboard.type / ... ）
   └─ core.py 在 dispatch 返回后，若 recording 开着且属操作类工具，append {tool,args,结果摘要}
recorder.stop()                      把 buffer 加工成 steps，调 skill_card.create 存成 draft
```

录制状态挂在 `set_run_session_context` / `get_run_session_context`（dispatch 已靠它跨进程
传上下文）这一层最自然。

**录制本身轻，把"裸动作流"加工成"好卡片"才是重点**，停止时做四件事：

1. **锚点提取**（对应 §2.1）：把 `mouse.click(x,y)` 的坐标尽量换成稳定锚点——命中点的
   UIA 控件、周围图像模板、相对位置。没有这步，录出来只是本机坐标宏，不能跨环境共享（§6.1）。
2. **脱敏**（§6.6）：扫描 `keyboard.type` 等，把疑似敏感值（密码框输入、token、路径里的
   用户名）转成参数槽 `{{...}}` 并标 `secret`，不固化落库。
3. **断言生成**：纯录制只知"做了什么"，不知"该变成什么样"。两种补法——录制时把每个动作
   **之后的活动窗口/屏幕状态**也抄一份生成弱断言；或录完让 AI 回看 buffer **补写**每步 assert
   （后者质量更高）。无断言则失败时无法定位到第几步（§3.2 / §4.3 的前提）。
4. **噪声过滤**：区分**操作类**工具（mouse/keyboard/window…进卡片）与**观察类**工具
   （screen.capture / vision…只作锚点提取的上下文，不单独成步）；允许 `recorder.stop` 丢弃
   末尾 N 步或标记试错步。

> 鼠标光标指示（§2.1）建议做成录制里的一等动作：录制中 AI 把光标移到目标上触发一次
> "记住此处特征"，拿到比截图猜更准的锚点。

### 4.2 重放（execute）

新增 endpoint 工具 `card.execute(card_id, params, version?)`，endpoint 本地循环执行：

1. 校验 preconditions，不满足直接拒绝（不瞎跑）
2. **作用域锁定**（§2.2）：按 `app_scope` 解析目标窗口（进程过滤 + 可选窗口裁剪/置顶）；
   目标应用不存在按 `on_missing` 处理（默认 halt，绝不去别的窗口找）。后续所有识别都限定在此作用域内
3. 逐步定位：先按 `region` 收窄、`disambiguate` 相对锚点消歧（§2.3），再按 `target.anchors`
   优先级匹配（UIA / 图像模板 / 坐标），命中即执行，不过 LLM
4. **歧义护栏**：执行点击前先数命中数。`=1` 才点；`≥2 且无法消歧` → 不点，halt 回传
   "歧义：N 个候选 + 截图" 退回 AI（§2.3 铁律二）；`=0` 进下一步
5. **锚点全失败（0 命中）** → 若 `vision_fallback.enabled`，该步退回 AI 视觉发现（截图 + hint
   重新找），找到后用新命中点回写锚点（自愈）；否则按 `on_fail` 处理
6. 每步执行后校验 assert，不过 → 按 `on_fail`（halt/retry/skip）
7. halt 时打包失败现场回传（见 4.3）
8. 全部通过 → 校验 postconditions → 更新统计

### 4.3 失败自愈（heal）

失败现场结构化回传，让 AI 精准改那一步并**从失败步续跑**，而非从头重来：

```jsonc
{
  "card_id": "card_a1b2c3",
  "version": 3,
  "failed_step": 4,
  "reason": "ambiguous",                  // not_found | ambiguous | assert_failed
  "matched_count": 3,                     // 歧义时回传命中数，供 AI 判断
  "candidates": [                         // 各候选的可区分信息（行文本/位置）
    { "row_text": "报告.docx", "bbox": [..] },
    { "row_text": "草稿.docx", "bbox": [..] }
  ],
  "expected": "该文件行被删除",
  "actual": "命中 3 个删除图标，无法确定点哪个",
  "screenshot": "data:image/png;base64,...",
  "resume_from": 4
}
```

AI 修改后：补一个消歧器（§2.3，如绑定唯一行文本）→ 写入新版本（version++）、记录
change_summary，从 `resume_from` 续跑验证。对 public 卡默认 fork 而非原地改（见 6.4）。

### 4.4 晋升与降级（status）

```
draft ──(auto 卡：supervised 下连续成功 N 次 / teach 卡：dry_run 通过即可)──▶ supervised → trusted
trusted ──(连续失败 M 次 / 跨到新环境)──▶ 回落 supervised
任意 ──(被标记问题 / 长期不用)──▶ deprecated
```

- **录制模式影响起点**：`teach` 卡的断言、消歧已被人当场确认过，dry_run 跑通即可直接进
  `supervised`；`auto` 卡断言/消歧靠猜，需更多观察期再晋升。
- `draft`：刚录制，每次跑都要 AI 盯着
- `supervised`：跑可以，但结果要 AI/人确认；统计成功率
- `trusted`：可在该环境自动重放，无需逐步监督
- `deprecated`：停用，仅保留历史

destructive 步骤即便在 trusted 卡里也要二次确认（见 6.6）。

### 5.1 桌面端锚点能力现状（决定第一版能做到多强）

录制/重放的质量天花板取决于桌面工具回传什么。按当前 `agent/windows/src/tools/` 实测：

| 能力 | 现状 | 对锚点/作用域的影响 |
| --- | --- | --- |
| `mouse.click` 回传 | 仅 `{success, button, position(坐标)}` | **没有 UIA 控件信息** → 第一版锚点只能用坐标 |
| `screen.capture` / `capture_region` | ✅ 可整屏 / 区域截图，返回 dataUrl | 图像模板锚点可行（截命中点周围一块）；按 rect 裁剪作用域可行 |
| `screen.info` | ✅ 返回屏幕尺寸 + 光标位置 | 鼠标指示法、environment_signature 可行 |
| `window.list` 回传 | ✅ `{hwnd, pid, title, name(进程名)}` | 作用域**进程过滤可行**（§2.2 锁定 QQ 排除微信） |
| `window.list` 窗口 rect | ❌ 不回传 x/y/w/h | 按窗口**像素裁剪截图暂不可做**，第一版退化为 `window.focus` 置顶后全屏识别（弱隔离） |
| `window.focus` | ✅ 按标题激活窗口 | 弱隔离：识别前先把目标窗口置顶 |
| UIA 控件查询 | ❌ 当前不存在 | `uia` 锚点需先给桌面端补能力 |

由此**录制分两步落地**：

- **第一版（能用、先跑通闭环）**：作用域 = 进程过滤 + `window.focus` 置顶（弱隔离）；
  锚点 = 周围**图像模板** + **坐标**兜底 + `vision_fallback`。跨环境共享弱（图像模板对
  分辨率/主题敏感），但本机重放 + 失败自愈闭环可完整跑通。
- **第二版（强隔离 + 可跨环境共享）**：给桌面 endpoint 增加三项回传——①`window.list` 补
  窗口 rect（按窗口物理裁剪截图，作用域硬隔离），②点击命中点的 UIA 控件名/类型，
  ③命中点周围图像块。补齐后作用域升到像素裁剪、`anchors` 升到"UIA 优先"，卡片才真正可跨机共享。

## 6. 共享带来的新坑与解决方案

以下六个坑是本地方案不存在、共享后才出现的，必须在 schema 与运行时同时解决。

### 6.1 环境可移植性（最致命）

**问题**：AI-A 录的卡到 AI-B 机器：路径 `D:\work` 不存在、"桌面第 3 个图标"位置不同、
软件版本不一、未登录。共享卡到别的机器一跑就废。

**解决**：
1. **全参数化**：路径、文件名、URL、账号全部转参数槽，禁止硬编码（录制时脱敏强制执行）。
2. **语义锚点替代位置**：见第 2 章定位策略，坐标仅作最后兜底。
3. **显式前置条件**：`preconditions` 声明 `requires: ["已安装 VSCode", "已登录 X"]`，
   重放前先校验，不满足直接拒绝并回传原因，而不是盲目执行。
4. **环境签名**：卡片记录 `environment_signature`，与目标环境差异过大时主动提示需复核。

### 6.2 权限必须按"调用方"收敛，卡片不能成为提权后门

**问题**：AI-A 有 shell 权限，录了跑 shell 的 trusted 卡共享出去；AI-B 没有 shell
权限却调用它。按谁的权限算？

**解决**：**调用方权限说了算**。
1. 卡片只声明 `capability`（需要哪些工具），不携带任何权限。
2. 重放前，运行时把 `card.capability` 与**调用方自己的** `effective_tool_allowlist`
   取交集（复用 `core.py` 现有逻辑）；缺任一所需工具即拒绝执行并说明缺哪项权限。
3. 这样共享技能库永远绕不过既有权限治理，不会成为提权洞。

### 6.3 信任分是"按环境"的，不是全局

**问题**：A 机器 99% 成功率的卡，到 B 机器可能 0%。全局 trusted 会给 B 一个"放心
自动跑"的假信号。

**解决**：
1. 统计以 `(card_id, environment_signature)` 为维度（见 3.4）。
2. 卡片被新环境/新 AI 首次调用时，信任分不继承，**回落 supervised**，重新积累。
3. 前端展示信任分时标注所属环境，不展示一个误导性的全局数字。

### 6.4 并发编辑 / 自愈冲突

**问题**：共享卡 + "出错返回 AI 改卡" = 多个 AI 可能同时改同一张卡。B 为适配自己环境
改的那步，可能正好搞坏 A 在用的版本。

**解决**：
1. **copy-on-heal / fork**：AI 改 `public` 卡时默认 fork 出自己的版本，不原地改；
   优质 fork 由 `Archivist` 合并回主卡。
2. **版本钉选**：调用方可 pin 版本（`card.execute(..., version=3)`），不被他人"修复"带飞。
3. **乐观锁**：写卡片带 `version` 校验，冲突则拒绝并要求基于最新版本重做。
4. 完整版本历史（3.3）支持回滚。

### 6.5 发现与去重

**问题**：库一大会出现 50 张叫"保存文件"的卡，变成垃圾抽屉。

**解决**：
1. **语义搜索**：用 embedding 对 name/description/domain 建索引（`librarian` + `web_search`）。
2. **去重提示**：新建卡片时检索相似卡，提示复用或合并而非新建。
3. **策展/晋升**：`Archivist` 负责把某场景的标准卡盖章为 canonical，其余降级或归并。
4. 按 surface/domain/scope 多维筛选（前端"沉淀技能"页）。

### 6.6 安全与隐私

**问题**：录制可能把敏感值（文件名、URL、密码）固化进卡片；一张恶意/有 bug 的破坏性
卡共享到 public 能批量毁很多机器。

**解决**：
1. **录制脱敏**：敏感输入转参数槽并标 `secret: true`，运行时不落库、不随卡片共享。
2. **共享范围分级**：`scope = private | team | public`，默认最小可见范围。
3. **破坏性卡审核**：含 destructive 步骤的卡共享到 `team/public` 前必须 `Archivist` 审核。
4. **destructive 二次确认**：即便在 trusted 卡，destructive 步骤执行前仍需确认。
5. **审计**：所有 `card.execute` 调用记录调用方、参数（脱敏后）、结果，纳入可观测链路
   （复用 README 的 `detected/parsed/authorized/executing/completed/failed` 状态机，按步追踪）。

## 7. 实施阶段建议

| 阶段 | 目标 | 状态 |
| --- | --- | --- |
| S1 数据地基 | 卡片 schema、三表、`skill_card` MCP 存取工具（含乐观锁/fork/按环境信任分） | ✅ 已落地 |

| S5 共享治理 | scope/权限交集/fork/审核；前端"沉淀技能"管理页 | |

介入（§2.1），常态走确定性锚点；语义锚点、参数槽、逐步断言、按环境信任分、权限契约、
共享范围在 S1 一次性设计到位，后续才稳。第一版锚点受桌面工具回传所限（§5.1），先用
图像模板 + 坐标跑通闭环，UIA 锚点随 S4.5 补齐后再升级为可跨环境共享。

# 第四章 浏览器 MCP 工具整理方案

# 浏览器 MCP 工具整理方案

## 背景

浏览器扩展（`agent/extension`）对外暴露的 `browser_*` MCP 工具增长到 **47 个**，
在工具列表、权限编辑器、AI 的 `list_tools` 里都显得「又多又杂」。本方案在不削减
任何实际能力的前提下，把工具收敛、分类统一，降低认知与维护成本。

## 问题诊断

1. **CRUD 类工具把列表撑大了。** 状态管理类操作按「资源 × 动作」拆成了一堆同质工具，
   占了将近一半：
   - `browser_cookie_list / get / set / delete`（4）
   - `browser_storage_get / set / remove / list`（4）
   - `browser_session_save / list / restore / delete`（4）
   - `browser_tab_list / open / close`（3）
   - `browser_history_back / forward`（2）
   - `browser_profile_info / set`（2）

   合计 **19 个**工具，本质是 6 组资源上的增删查改。

2. **分类逻辑在 3 处各写了一份。** 工具该归到哪个「浏览器 X」分组，分别硬编码在：
   - `agent/extension/src/lib/tools/definitions.ts`（数组顺序隐含分组）
   - `web/src/utils/mcpTools.ts`（`getEndpointCapabilityTag`）
   - `web/src/components/dashboard/modals/AgentMcpScopeEditor.vue`（`browserIntro`）

   三处各自维护一份 `browser_*` 名单，加新工具要改三遍，且已经出现不一致
   （Web 端把 storage 全家归到「浏览器数据」，权限编辑器又单独分了 session / cookie / profile）。

3. **`browser.ts` 918 行**把 helper、47 个实现、路由揉在一起，可读性差。

## 整理目标

- **工具数 47 → 34**：把上面 19 个 CRUD 工具收敛为 6 个带 `action` 参数的工具。
- **能力不减**：每个被合并的旧工具，都能用「新工具 + action」一一对应地表达。
- **分类单一来源**：在 `definitions.ts` 用一份 `BROWSER_TOOL_CATEGORIES` 描述分组，
  扩展端的 popup 直接用它；服务端 / Web 端按统一前缀规则归类，去掉重复名单。
- **向后兼容**：旧工具名（`browser_cookie_get` 等）仍可被调用，路由层把它们改写为
  「新工具 + action」。已保存的权限 scope 不会因为改名而报错——离线/在线能力按名取交集，
  旧名只是不再出现在能力列表里。

## 合并设计

| 新工具 | 合并的旧工具 | `action` 取值 |
| --- | --- | --- |
| `browser_tab` | `tab_list` / `tab_open` / `tab_close` | `list` / `open` / `close` |
| `browser_history` | `history_back` / `history_forward` | `back` / `forward` |
| `browser_cookie` | `cookie_list` / `cookie_get` / `cookie_set` / `cookie_delete` | `list` / `get` / `set` / `delete` |
| `browser_storage` | `storage_get` / `storage_set` / `storage_remove` / `storage_list` | `get` / `set` / `remove` / `list` |
| `browser_session` | `session_save` / `session_list` / `session_restore` / `session_delete` | `save` / `list` / `restore` / `delete` |
| `browser_profile` | `profile_info` / `profile_set` | `info` / `set` |

合并工具的其余参数沿用原工具（如 `cookie` 的 `name/value/domain/...`、`storage` 的
`key/value/type/prefix/...`），只是多了一个必填的 `action` 来选动作。

**保持独立**的核心操作不动（导航、点击、输入、截图、滚动等高频且语义清晰的工具
拆开更利于 AI 理解），避免过度合并。

## 分类（5 大类，34 工具）

| 分类 | 工具 |
| --- | --- |
| 导航与搜索 | `navigate` `search` `history` |
| 页面观察 | `screenshot` `get_content` `dom_snapshot` `page_info` `find_text` `find_popups` `performance` `network_log` `iframe_list` |
| 页面交互 | `click` `double_click` `right_click` `type` `press_key` `hover` `scroll` `wait` `drag` `fill_form` `select` `close_popup` |
| 数据与脚本 | `evaluate` `extract` `clipboard_write` `file_upload` `download` |
| 浏览器状态 | `tab` `cookie` `storage` `session` `profile` |

## 改动清单

| 文件 | 改动 |
| --- | --- |
| `agent/extension/src/lib/tools/definitions.ts` | 按分类重排；新增 6 个合并工具的 schema；导出 `BROWSER_TOOL_CATEGORIES` |
| `agent/extension/src/lib/tools/browser.ts` | 新增 6 个按 `action` 分发的 handler；旧实现保留为内部函数；路由表用「新工具 + 旧名别名」 |
| `agent/extension/src/lib/tools/executor.ts` | `inferTool` 里 `browser_tab_list` → `browser_tab` |
| `agent/extension/src/popup/mcp.ts` | 工具列表按 `BROWSER_TOOL_CATEGORIES` 分组展示 |
| `web/src/utils/mcpTools.ts` | `browser_*` 改为按统一前缀归类，去掉冗长名单 |
| `web/src/components/dashboard/modals/AgentMcpScopeEditor.vue` | `browserIntro` 改为按新分类，删冗余名单 |
| `README` / `doc/prompt/浏览器助手prompt.md` | 同步工具说明 |

## 兼容性与风险

- **AI 调用**：模型每轮通过 `list_tools` 拿到的是新工具表，会用新名；少量历史习惯
  用旧名的调用由路由别名兜底。
- **已保存权限**：scope 以工具名取交集，旧名从能力表移除后自动失效，不会报错；
  用户重开权限编辑器即可看到收敛后的新工具并勾选。
- **构建**：扩展用 esbuild 打包，Web 用 Vite + tsc，改动均为同语言内重构，类型可校验。

# 第五章 Claude Code 集成-权限分层设计

# Claude Code 集成 - 权限分层设计

本文记录在 agent 端(Electron 桌面端)集成 Claude Code CLI 时的权限模型。
权限是整套方案中最复杂的一环,需要在动手前先达成共识。

## 1. 背景与边界

### 1.1 集成形态

- Claude Code CLI 嵌入 `agent/windows`(后续扩展 `agent/linux`),不进服务器
- Electron 主进程通过 `child_process` 起子进程:
  `claude --print --output-format stream-json --input-format stream-json ...`
- API key:MVP 由用户自填(`electron-store` 加密存储),后续支持服务器发放
- 工具暴露:本地 stdio MCP server 只补 Claude 没有的能力(键鼠模拟、屏幕捕获、
  窗口管理),Read/Write/Bash 沿用 Claude 自带
- 任务触发:用户在 agent UI 主动发起,或由 `connector_runtime` 下发任务到 agent

### 1.2 Skill 来源

- 当前:服务器下发(可信)
- 后续:Claude Skills 官方仓库、ClowHub(待确认完整名称)、用户本地
- 三种来源信任级别不同,权限策略不同

### 1.3 为什么权限要专门设计

三类风险面:

1. **CLI 默认权限模式不适配后端化的使用方式**
   `default` 模式每次工具调用都问用户,会卡住自动任务流;`bypassPermissions`
   一刀切又放弃了所有边界。需要按场景分别配置。
2. **服务器下发任务是新增的攻击面**
   即使可信服务器,被劫持的下发通道也可能让 Claude 在用户机器上做超出预期的事。
3. **第三方 skill 等价于远程代码下发**
   skill 文件包含可执行 bash / 脚本片段,启用即执行,等同于 RCE,
   不能与本地用户编写的 skill 一视同仁。

## 2. 权限分层

四层从外到内,任一层拒绝即整个调用拒绝。

### 2.1 第一层:CLI 权限模式

按 session / task 场景选择 `--permission-mode`:

| 场景 | 模式 | 说明 |
|---|---|---|
| 普通对话(用户在 UI 自己发起) | `acceptEdits` | 文件编辑自动批准,其他工具询问(由第三层 hook 接管询问) |
| 服务器下发的明确任务 | `bypassPermissions` | 任务包内已声明 allowedTools,放权给后续层 |
| 用户标记为"高风险"或预览模式 | `plan` | 只规划不执行,生成可读步骤,用户确认后再切真执行 |

**不允许** 全局默认 `bypassPermissions`。

### 2.2 第二层:工具白名单 / 黑名单

每个 session 和 task 独立配置:

- `--allowedTools` 限定本次 session 能用的工具集合
- `--disallowedTools` 显式禁用危险工具(`Bash(rm:*)`, `Bash(sudo:*)`, ...)
- 服务器下发任务时,任务包自带白名单;本地用户发起的对话用 agent 设置里的默认白名单

工具命名遵循 Claude Code 规范:
- 内置:`Read`, `Write`, `Edit`, `Bash`, `Glob`, `Grep` ...
- MCP:`mcp__<server>__<tool>`,例如 `mcp__heysure-local__screenshot`

### 2.3 第三层:运行时 hook 拦截 + UI 确认

Claude Code 支持 `PreToolUse` / `PostToolUse` hook。Electron 主进程注册 hook:

- `PreToolUse` 触发 → 主进程根据规则决定 `allow / deny / ask`
- `ask` 路径推 renderer 弹窗,展示工具名、参数摘要、风险标签
- 用户选项:`本次允许` / `本次拒绝` / `始终允许此工具` / `始终拒绝此工具`
- 偏好持久化到 `electron-store`,可在 agent 设置里清空

**必备 UI 元素:**

- 全局"紧急停止"按钮(任何窗口可见),按下立即 SIGTERM 所有 Claude 子进程
- 实时执行指示器:"Claude 正在调用 `mcp__heysure-local__click`(参数:...)"
- 历史决策面板:能看到 / 撤销曾经的"始终允许"决定

### 2.4 第四层:本地 MCP server 自身权限

MCP server(`services/localMcpServer`)在 tool handler 里再做一层校验:

| 工具 | 权限点 |
|---|---|
| `screenshot` | 首次调用走 OS 级权限弹窗(Win 直接通过,mac 需 ScreenCapture 授权,Linux Wayland 受限);维持"截屏总开关" |
| `click / type / key` | 没有 OS 弹窗,MCP server 维持"键鼠操作总开关",默认关,renderer 切换;每次调用记录到操作日志 |
| `fs.read / fs.write`(若启用) | 限定在用户授权目录,默认仅 `userData/`,白名单外的路径直接拒绝 |
| `window.activate` | 同键鼠总开关 |

第四层与第三层的关系:第三层是 Claude 调用进入 MCP server **之前** 的拦截,
第四层是 MCP server 自身的硬边界。任一拒绝即失败。

## 3. 服务器下发任务的权限边界

这是 agent 路线引入的新攻击面,**单独制定策略**。

### 3.1 任务下发协议

`connector_runtime` 下发的 `claude-code` 类型任务必须携带权限策略包:

```jsonc
{
  "task_type": "claude-code",
  "task_id": "...",
  "prompt": "...",
  "permission_policy": {
    "permission_mode": "bypassPermissions" | "acceptEdits" | "plan",
    "allowed_tools": ["Read", "Bash(git:*)", "mcp__heysure-local__screenshot"],
    "disallowed_tools": ["Bash(rm:*)"],
    "max_steps": 20,
    "workdir": "userData/tasks/<task_id>/",
    "require_user_confirmation": ["mcp__heysure-local__click", "mcp__heysure-local__type"],
    "timeout_seconds": 600
  }
}
```

### 3.2 agent 本地硬上限

agent 维护一份"服务器最多能让我干什么"的本地策略文件
(`userData/server-task-policy.json`),用户可在设置里编辑。
服务器策略与本地硬上限取交集 — 服务器无法突破本地上限。

默认硬上限示例:
- 永远禁用 `Bash(sudo:*)`, `Bash(rm:-rf*)`, 任意网络下载执行
- 键鼠 / 截屏始终需要用户确认(`require_user_confirmation` 不可被服务器关掉)
- `workdir` 必须落在 `userData/tasks/` 子目录,不允许跳出

### 3.3 task 隔离

每个服务器任务一个临时 workdir,跑完清理。生成的 `.mcp.json` 也在 workdir 内,
不污染用户的常规对话 session。

### 3.4 task 结果回传

回传内容:最终 assistant 消息、工具调用列表、退出原因、token 用量。
**默认不回传** thinking 块全文(隐私 + 带宽),仅回传摘要。
用户可在设置里改成"回传完整 thinking"。

## 4. Skill 仓库权限

skill 文件包含可执行内容,需要按来源分级。

### 4.1 来源分级

| 来源 | 信任等级 | 默认策略 |
|---|---|---|
| 用户本地编写(`userData/.claude/skills/local/`) | 高 | 直接启用 |
| 服务器下发(`userData/.claude/skills/server/`) | 中 | 启用前列出 skill 名 + 摘要,用户一次性确认本服务器 |
| Claude Skills 官方仓库 | 中 | 同上,且锁定到具体 release 版本 |
| ClowHub / 其他第三方仓库 | 低 | 每个 skill 独立确认,默认 `plan` 模式预览,工具白名单严格 |

### 4.2 启用流程

1. 用户在 agent UI"Skill 市场"页选中要装的 skill
2. 拉取到 cache 目录,展示 SKILL.md 全文 + 声明使用的工具列表
3. 用户确认 → 写入 `userData/.claude/skills/<source>/<skill-name>/`
4. 注入到 session 的工具白名单中(白名单与 skill 声明的工具取交集)

### 4.3 skill 沙箱

- 低信任来源的 skill 默认运行在 `plan` 模式预览,用户看完计划再切真执行
- skill 声明的 `bash` 命令走严格白名单匹配,不允许任意 `Bash`
- skill 不允许声明 `mcp__heysure-local__*` 工具(本地敏感工具只能由 agent 自身配置授权)

### 4.4 同名冲突

多源同名 skill 按 `本地 > 服务器 > 官方 > 第三方` 优先级,UI 给出冲突提示。

## 5. 思考(extended thinking)的处理

thinking 不是权限问题,但与服务器下发/回传策略相关,放这里统一记录。

| 决策 | 默认 |
|---|---|
| 默认是否开启 | 关 |
| 用户对话开关 | 是,renderer 一个 toggle |
| 任务级开关 | 服务器下发的任务可在 `permission_policy` 里要求开启 |
| 回传服务器 | 否,只回传 assistant 文本与工具调用 |
| renderer 展示 | 折叠,可展开 |

## 6. 跨平台考量

- robotjs 在 Linux Wayland 下基本不工作,需要 `xdotool` (X11) 或 `ydotool` (Wayland, 需 root) 兜底
- 截屏 Linux 也按显示协议分流:`gnome-screenshot` / `grim`
- 上述差异封装在本地 MCP server 内,Claude 看到的工具签名跨平台一致
- 文件路径全部用 Node `path` 模块,不写死分隔符

## 7. 能力分发与知识库对接

`librarian`(图书管理员)知识库已经为后续沉淀预留了两个内置栏目,Claude Code
集成方案直接复用它们作为"能力包"的载体,不再造一套知识管理。

### 7.1 现有栏目与新概念的对齐

| 知识库栏目(title) | 当前 summary | 在本方案里承载 | 文件形态 |
|---|---|---|---|
| **传承思想** | 预留沉淀 Markdown 思想文档 | **Skill**(Claude Code 的 SKILL.md + 资源) | `<skill>/SKILL.md` (+ 资源目录) |
| **传承技能** | 预留沉淀 Python 脚本技能 | **Tool**(MCP tool 定义 + 实现) | `<tool>/tool.json` + 脚本 |

> 注:`server/api/services/librarian_service.py` 里目前的变量命名是反的 —
> `builtin.inheritance_skills` 实际对应 title "传承技能"(=Tool),
> `builtin.inheritance_tools` 实际对应 title "传承思想"(=Skill)。
> 建议落地新分栏时把变量名与 title 对齐,或改为
> `builtin.inheritance_thoughts` / `builtin.inheritance_skills`。
> 这是一次性破坏改动,改完语义清晰。

### 7.2 "传承思想"(Skill)分栏

| 子分栏 | 来源 | 落盘路径 | 信任档 |
|---|---|---|---|
| **沉淀 skills** | 用户在本机创作 + AI 间转授积累 | `userData/.claude/skills/local/<skill_id>/` | 高 |
| **网络互联 skills** | Claude Skills 官方仓库 / ClowHub / 其他源 | `userData/.claude/skills/remote/<source>/<skill_id>/` | 低/中,沿用 §4 来源分级 |

UI 上"传承思想"展开两个 tab,数据层用 `subcategory` 字段(`local` / `remote`)区分,
拉取/同步/缓存/权限策略按 §4 分别处理。前端入口复用
`web/src/components/dashboard/panels/KnowledgeBasePanel.vue`。

### 7.3 "传承技能"(Tool)分栏

与思想对齐,同样两个子分栏:

| 子分栏 | 来源 | 落盘路径 |
|---|---|---|
| **沉淀 tools** | 用户/AI 在本机创作的 MCP tool 包 | `userData/mcp-tools/local/<tool_id>/` |
| **网络互联 tools** | 从外部源拉取的 MCP tool 包 | `userData/mcp-tools/remote/<source>/<tool_id>/` |

Tool 风险高于 skill(本身是可执行代码,Claude 会自动调用),
**网络互联 tools 默认不允许 AI 间转授,只能由用户主动装载并确认源**。

### 7.4 统一抽象:能力包(Capability)

skill 与 tool 用同一个 `Capability` 抽象表示,差异在 `kind`:

```jsonc
{
  "capability_id": "skill.web-summarize.v3",
  "kind": "skill" | "tool",
  "subcategory": "local" | "remote",
  "source": "user" | "ai:<ai_id>" | "remote:claude-skills" | "remote:clowhub" | ...,
  "title": "网页摘要",
  "summary": "...",
  "version": "3.0.1",
  "manifest": { /* SKILL.md frontmatter 或 tool.json */ },
  "files": [...],
  "declared_tools": ["Read", "WebFetch"],         // skill 声明使用的工具
  "provides_tools": ["mcp__local__summarize"],    // tool 类型才有
  "trust_tier": "high" | "medium" | "low",
  "transferable_by_ai": true | false              // 是否允许 AI 间转授,见 §7.6
}
```

librarian 现有"流程知识"记录可直接扩展这些字段(`subcategory`、`kind`、
`trust_tier`、`transferable_by_ai`),不必新建表。

### 7.5 装载与卸载(用户 → AI)

每个 AI 配置(`ai_config`)新增两个关联字段:

- `installed_skills`: capability_id 列表
- `installed_tools`: capability_id 列表

UI 操作:
- 在"传承思想 / 传承技能"列表里选中能力包 → 选目标 AI → 点"装载到此 AI"
- 在 AI 详情页"已装载能力"区域单独卸载

后端动作:
- 装载:写 `ai_config.installed_*`,在该 AI 下次启动 Claude Code session 时生效
- 卸载:反向

**装载不等于立即生效。** Claude CLI 启动后不热加载 `.mcp.json` 与 skills 目录,
运行中的 session 看不到新装的能力,需要新会话生效。UI 必须给出
"已装载,新会话生效"提示。

### 7.6 AI ↔ AI 转授

发起方 AI 通过新增 MCP tool 把自己已装载的能力授予另一个 AI(实现位置
`server/api/mcp/tools/`,与现有 `ai.send_message` 同层):

- `capability.grant(target_ai_id, capability_id, reason)`
- `capability.revoke(target_ai_id, capability_id)`

**约束(关键安全设计):**

1. 只能转授 `transferable_by_ai = true` 的能力包
2. 默认值:
   - 沉淀来源(`subcategory=local`) → true
   - 网络互联来源(`subcategory=remote`) → false
   - tool 比 skill 更严:网络互联 tool 强制 false,用户也不能改
3. 受授方默认不自动接受,生成一条 pending grant,等用户在 UI 确认
   - 用户可对 `<发起 AI, 受授 AI, 能力分类>` 三元组设白名单跳过确认
   - 跨用户场景(多用户)的转授永远需要确认,不允许白名单
4. 转授审计:新增 `capability_grant_log` 表,记录
   `from_ai / to_ai / capability_id / ts / reason / decision`,UI 可查

**为什么要这么严?**
AI 间转授等价于"一个 AI 替另一个 AI 决定信任新能力"。如果不卡,
一个被劫持或越权运行的 AI 可以把恶意远程能力批量授予整个数字社会的其他 AI,
权限边界形同虚设。

### 7.7 运行时注入:session 启动时构建工具/技能视图

每次 agent 端启动 Claude Code 子进程时,基于目标 AI 的装载清单,在 workdir
内动态构建:

- `.mcp.json` — 由 `installed_tools` 中的 tool 包合并 + 本地系统 MCP server
  (键鼠/截屏)生成
- `.claude/skills/` — `installed_skills` 中的 skill 文件软链接进来(Linux/mac)
  或拷贝(Windows)
- `--allowedTools` / `--disallowedTools` — 从 capability 的 `declared_tools` /
  `provides_tools` 汇总,与 §3 服务器策略包取交集

session 结束清理 workdir,装载状态长期保存在 `ai_config`,下次重建。

### 7.8 与 §4 Skill 仓库权限的关系

§4 关注"skill 从哪儿来 / 启用前怎么审"(入库审查)。§7 关注"能力包怎么分发到
具体 AI"(装载 / 转授)。两者层叠:

1. 先按 §4 决定 skill / tool 是否进入"传承思想 / 传承技能"知识库
2. 再按 §7 决定能力包装到哪个 AI 上

两层都通过,某个 AI 才能在 session 里真正用上。

## 8. 实施路线

### MVP(最小可用)
- CLI 嵌入打包(electron-builder extraResources)
- 用户自填 API key,electron-store 存储
- `acceptEdits` 模式 + 紧急停止按钮 + 简单工具白名单
- 本地 MCP server 暴露 `screenshot` + `click` + `type`(总开关默认关)
- 仅支持用户在 UI 主动发起的对话

### v2
- 第三层 PreToolUse hook 接入,弹窗确认 + 偏好持久化
- `connector_runtime` 下发任务 + 权限策略包 + 本地硬上限
- 服务器下发 skill 通道
- `librarian` 栏目变量名与 title 对齐;"传承思想 / 传承技能"分别落地
  `local` / `remote` 子分栏字段
- `Capability` 抽象在数据层落地;`ai_config.installed_skills` /
  `installed_tools` 字段及"装载到此 AI"UI
- session 启动时的 `.mcp.json` / `.claude/skills/` 动态注入(§7.7)

### v3
- 接 Claude Skills 官方仓库,版本锁定
- 第三方 skill 仓库(ClowHub 等)对接,plan 预览
- skill 同名冲突 / 灰度发布
- AI ↔ AI 转授(`capability.grant` / `capability.revoke`)+ pending 确认 UI +
  `capability_grant_log` 审计
- 网络互联 tool 子分栏开放(默认禁止 AI 转授,见 §7.6)

### v4(可选)
- 服务器统一下发 API key(平台计费模式)
- 多 session 并行 / 后台任务队列
- Linux agent 跨发行版打包
