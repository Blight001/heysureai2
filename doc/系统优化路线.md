# 系统优化路线

本文记录当前系统在进入 skill / hook 体系之前应优先完成的工程优化。目标是先降低聊天运行时和 MCP 执行链路的复杂度，让后续扩展有稳定边界。

## 优先级

1. MCP 文本解析集中化
   - 状态：已完成第一阶段
   - 目标：后端只保留一个 MCP block 正则和 payload 解析入口，避免 `chat_worker`、飞书通知、前端、扩展各自维护不一致规则。
   - 交付：`api/chat_runtime/mcp_parser.py`、基础单元测试、后端调用方切换、Web 展示层正则集中化。

2. MCP 执行状态可观测性
   - 状态：待开始
   - 目标：把 `detected / parsed / authorized / executing / completed / failed` 等阶段记录到统一结构，便于定位“模型写了但没执行”的问题。
   - 交付：工具调用状态对象、失败原因分类、UI 可读详情。

3. Prompt 组装分层
   - 状态：待开始
   - 目标：将全局 MCP prompt、任务 prompt、AI 间通信 prompt、自动控制 prompt 分段构建，减少重复和冲突。
   - 交付：`prompt_builder` 模块、最终 prompt 分段预览。

4. 工具权限运行时分层
   - 状态：待开始
   - 目标：区分 AI 配置层的理论可用工具和本轮上下文实际允许工具，为后续 hook/skill 留出安全边界。
   - 交付：`available_tools` 与 `runtime_allowed_tools` 合并规则。

5. 运行态数据与仓库隔离
   - 状态：待开始
   - 目标：减少 `server/api/data`、`server/data` 中运行态 JSON 的 git 噪声。
   - 交付：模板文件策略、`.gitignore` 调整或初始化迁移脚本。

6. 前端与扩展 dist 策略
   - 状态：待开始
   - 目标：明确 `web/dist` 和 `agent/extension/dist` 是否作为部署产物提交，减少 review 噪声。
   - 交付：构建产物管理约定。
