# 进化流程设计（自进化执行闭环）

> 本文记录 HeySure AI 2.0 "知识 + 工具"自进化闭环的设计思路：先复述原始构想，
> 再对照当前代码标出已实现 / 待实现的缺口，给出推荐的增强版进化飞轮与落地优先级，
> 并对照 2025–2026 自进化 agent 前沿（含"误进化"风险）校准设计（见第五节）。
> 角色与知识库现状见 [`角色与知识流转.md`](角色与知识流转.md)。

## 一、核心思路：一条自进化执行闭环

围绕执行体（图中代号「贝」，落地为数字成员 Agent）的一条闭环：

```
执行 → 总结 → 沉淀知识 → 复用 → 进化工具
```

让执行体把"怎么做成的 / 踩了什么坑 / 可复用的工作流"沉淀成**知识**和可演进的
**MCP 工具**，下次自动复用、自动纠错。原始构想（手绘草图）包含四块：

- **传承知识（知识库）**：设备端分类、新建、编辑、删除、读取（列表/详情）、联网下载。
- **角色与数据流**：`知`(知识) / `网`(联网) / `贝`(执行体) / `总`(总结) / `M`(模型) / `工`(工具)
  之间流转；贝产出三类东西——**遇到的报错 / 完整的工作流 / 可简化的工作流**。
- **工作流**：工作流可被「执行」。
- **传承工具（MCP）**：读取(列表/详情)、新建、编辑、删除、**版本(列表/详情/撤回)**；
  并要求 ① 动态加载 MCP ② 主动修改 MCP 代码 ③ 版本回溯。

四条闭环规则（草图右上）：

1. 贝任务执行后对用到的 MCP **主动总结**。
2. 阿（核心管理员/阿尔法）把贝的总结**识别、沉淀成知识库**。
3. 贝**读知识库**来决定 MCP 调用。
4. 贝**读 MCP 库知识自动跑工作流**，并**主动处理报错、再次总结**。

| 角色（贝/阿/总） | 有对应角色，但"贝/阿/总"是叙事代号，落地为 `assistant_admin`/`digital_member`/`is_librarian` | `server/api/models/ai_config.py`、`角色与知识流转.md` |

### 待实现 / 半实现（右半边为主，按缺口大小排序）

1. **传承工具（MCP）的增删改** —— 现在只能"读"（列表/describe），不能在产品里
   新建/编辑/删除工具。工具写死在 `server/mcp_runtime/mcp/tools/*.py`，靠热重载生效。**❌**
2. **MCP 版本管理（列表/详情/撤回/回溯）** —— registry 只有自增 `.version` 字段，
   无历史、无回滚。`server/mcp_runtime/mcp/core.py`。**❌**
3. **运行时主动改 MCP 代码** —— 只有"传承思想"的 SKILL.md 能按行编辑，
   MCP 工具本体代码不能在运行时被 AI 改。**❌**
4. **报错自愈 + 读 MCP 知识自动编排工作流** —— 任务出错只把 `AITaskJob.status="error"`
   落库，无自动重试/自愈，也没有工作流编排引擎。**❌**
5. **执行后自动总结 MCP + 总结自动入库** —— 现在需 AI 显式调 `task.complete/inherit`
   传 summary（非执行完自动），且总结对象是任务而非"用到的 MCP 工具本身"；
   入库还要人审。**⚠️ 半实现**
6. **真正的工作流引擎** —— 只有任务 + 定时调度（queued→running→completed，可循环），
   没有 DAG / 分步管线 / "完整工作流 vs 可简化工作流"的表示与执行。**⚠️ 半实现**

**一句话**：左边"知识沉淀与复用"较完整；右边"把 MCP 工具当成可被 AI 增删改、
可版本回溯、可自动编排执行的活资产"基本空白。

## 三、推荐的增强版进化飞轮

相比原始草图，建议增加这几样东西：**① 去重/合并闸 ② 风险分级的审批门禁
③ 知识→工具的晋升通道 ④ 用真实轨迹做回归样例的沙箱 ⑤ 度量驱动的淘汰/降权（飞轮的收缩侧）
⑥ 任务内"反思重试"短回路（区别于任务间长回路）⑦ 定期对整个系统跑安全回归**。

```
              ┌──── 任务内短回路：失败→口头反思→当场重试(不落库) ────┐
              ▼                                                     │
执行(贝)  ──采集调用轨迹(后端结构化)──►  自动初稿总结(模型只做提炼)    │
   │           └─────────────────(仍失败/确认可复用才进长回路)──────┘
   │◄──── 复用 ────┐                              ▼
   ▼               │                     去重/合并  ──►  draft 知识
读知识库决定调用     │                              │
   │               │                  风险分级门禁(低险自动/高险人审)
   ▼               │                              ▼
报错? ──是──► 反思重试(有上限) ──仍失败──► 升级给人   active 知识 ──度量低于阈值──► 归档/淘汰
   │                                              │ 仅晋升"确定性子流程"
   └──否──► 成功                      被高频复用? ──是──►  晋升为 MCP 工具(draft)
   │                                               │              │
   └─►(轨迹沉淀为回归样例)───────────────────────────┘     沙箱用真实轨迹回放自测 + 权限审查
                                                                  ▼
                                          带成功率/使用统计   active 工具(可版本回溯) ──低成功率──► 降权/归档

   ┌─ 横切：每次进化后对【整个已进化系统】跑安全/红队回归用例集；分数下降→冻结进化+告警 ─┐
   └────────────────────────── 新工具先灰度到产出者本人，回归通过再放全员 ──────────────────┘
```

### 设计要点

**1. "知识"和"工具"是两层，有晋升关系，但只晋升"确定性子流程"**
- 知识 = markdown SOP，软的、给模型读的"怎么做"；工具 = 可执行 MCP 代码，硬的、确定性的。
- 进化的关键：某条 SOP 被反复使用 → **晋升**为一个 MCP 工具（把"每次让模型照着做"
  固化成"一次调用必成功"）。应显式加一条 **知识 → 工具 晋升通道**，而非两条独立 CRUD。
- ⚠️ **不是所有知识都该晋升**：要区分**确定性步骤**（如"调某 API、拼某固定格式"，可固化）
  与**判断性步骤**（如"看情况决定下一步"，必须留给模型）。把判断逻辑硬编码 = 工具变脆、
  一遇新场景就废。原则：**只晋升确定性子流程，判断逻辑留在知识层**。
- 晋升要有**可观测信号 + 阈值**（如：被复用 ≥N 次且成功率 ≥X%），而非凭感觉"高频"。

**2. 门禁按"风险"分级，而不是按"是不是 AI 生成"一刀切**
自进化下若每条知识、每个工具都人审，人会被淹没、飞轮转不起来。门禁维度应是**风险**：
- **低风险**（纯读、无副作用、不升级权限）→ 自动通过 / 事后抽审；
- **高风险**（写操作、改权限、覆盖已有工具）→ 强制人审（或更高角色 AI 审）。
- 复用现成模式 `librarian.propose → 审批 → active`：AI 只能提交到 **draft/灰度**，
  不能直接热加载进生产 registry；审批通过才 active。这把草图右上②（阿识别）与
  右下②（改代码）接起来：**阿/管理员当高风险审批闸**。

**3. 沙箱自测用"真实轨迹回放"做样例，并审查权限声明（不只看代码能不能跑）**
- 自测样例**不能让 AI 自己写自己测**（自己判卷无意义）。正解：**从真实执行轨迹里抽
  input/output 当回归用例**——工具改版后用历史真实调用重放，结果一致才算通过。
  这条让"调用轨迹采集"与"工具晋升/改版"咬合成一个闭环。
- 🔒 **安全底线**：AI 改 MCP 代码 = 可能给自己提权（写一个"能执行任意命令"的工具）。
  门禁审查的重点是**工具的权限声明 / 它要碰什么**（文件、网络、命令、DB），
  而不仅是"代码能跑通"。权限升级一律走高风险人审。

**4. "自动总结"不要做成"执行完调一次大模型凭记忆复述"**
- 总结**原料**由后端结构化采集（用了哪些 MCP、入参/出参、报错栈、耗时、最终成败），
  模型只负责"提炼成 SOP"。现有 `task.inherit` 已在自动抓 unfinished/artifacts/token，
  把这套扩展到"MCP 调用轨迹"即可，更省钱也更准。
- 入库前做 **去重/合并**（同 trigger 命中已有条目则 merge，不新建），防"知识污染"。
- 报错自愈要有 **重试上限 + 失败升级给人**，不能无限自循环（防"自激震荡"）。

**5. 版本回溯的边界：只回滚"定义"，不回滚"副作用"**
- MCP 工具若有副作用（写库、发消息、改文件），**代码回滚 ≠ 效果回滚**。
- 回溯只还原**工具定义/代码**到某历史版本；**已产生的副作用不在回溯范围**。
  UI/文档要讲清，避免用户误以为"回滚 = 撤销一切"。

**6. 分两层：任务内"反思重试" vs 任务间"沉淀入库"（防过度固化）**
- 借鉴 Reflexion / Voyager：失败时**先在同一任务内做口头反思 → 当场重试**（短回路，
  不落长期记忆）；只有**反复出现、确认可复用**的经验才进入任务间的知识库（长回路）。
- 现状的"报错就想往库里写"会把一次性偶发问题固化，加剧知识污染。**短回路优先，
  长回路要门槛**。

**7. 把 prompt / 策略也当成可进化面（最低成本的增量）**
- 自进化综述把可进化面分为五类：模型 / 记忆 / 工具 / **prompt 策略** / 工作流。
  我们目前只动了记忆(知识)+工具，漏了 prompt。
- 项目 prompt 本就存在 DB（运行时 prompt 在库里，`doc/prompt/*.md` 为种子），
  让管理员/系统**按度量回归来微调角色策略文本**几乎零成本，且高杠杆。
  改动同样走"draft → 灰度 → 审批 → active + 可回溯"，与工具一致。

**8. 安全是"持续回归"，不只是"单次门禁"**
- 前沿实证（misevolution，见第六节）：**即使每次改动都良性，记忆/工具累积后整体安全
  对齐仍会退化**。逐条权限审查（要点 3）覆盖不了这种"累积漂移"。
- 因此除 per-change 门禁外，需对**整个已进化系统**定期跑一套**安全/红队回归用例集**：
  固定一批"不该照做"的诱导样例，每次进化后重跑，分数下降即冻结进化并告警。
- 多 agent 共享同一知识/工具库时还有**互相投毒**风险：一个「贝」产出的坏工具会污染
  其他成员。新工具应**先灰度到产出者自己**，回归通过再放给全员（所有权 + 灰度隔离）。

## 三·补. 度量与淘汰（飞轮的收缩侧）

只往里加、不往外删，跑久了知识库和工具库会膨胀、过时、互相冲突。必须有**收缩机制**，
且收缩要**由度量驱动**——否则无法判断"自进化"是在变好还是变差。

**北极星 / 可观测指标**（同时是降权与淘汰的触发依据）：
- **任务成功率**（趋势上升才说明进化有效）；
- **人工介入率**（应随知识/工具成熟而下降）；
- **平均完成步数 / 耗时**（同类任务应越来越短）；
- 每条知识/工具的**使用次数、最近使用时间、成功率**。

**淘汰 / 降权路径**：
- 低成功率 → 自动降权（检索/调用优先级下调）或触发复审；
- 长期未用 / 被新工具取代 → 自动归档（移出默认检索，留痕可恢复）；
- 与现有"图书管理员 archive"闭环对齐，工具侧补一套等价的归档/降权。

## 四、落地优先级

1. **传承工具 CRUD + 版本管理/回溯** —— 缺口最大、且不碰危险的"AI 自动改代码"，
   纯人工管理 MCP 也立刻有价值。先做（注意标清版本回溯只回滚定义、不回滚副作用）。
2. **MCP 调用轨迹采集 + 自动初稿总结** —— 复用现有 `task.inherit` 钩子；
   轨迹同时沉淀为"回归样例"，供第 3 项沙箱回放复用。
3. **AI 自动建/改工具（风险分级门禁 + 沙箱回放 + 权限审查）+ 报错自愈** ——
   风险最高，放最后，且必须带门禁与提权审查。
4. **度量与淘汰（横切）** —— 越早埋点越好：先把使用次数/成功率/人工介入率等指标采起来，
   它既是淘汰/降权的触发依据，也是验证"自进化是否真在变好"的唯一手段。
5. **安全回归用例集（横切，越早越好）** —— 维护一批"不该照做"的红队样例，每次进化后重跑；
   这是防"累积安全漂移"的唯一手段，不能等功能都做完再补。
6. **prompt / 策略进化** —— 性价比最高的增量，项目 prompt 已在 DB，可在 1~2 项之后顺手接入。

## 五、与前沿的对照与"误进化"风险

调研了 2025–2026 的自进化 agent 前沿后，本设计的定位与仍存的风险如下。

**与前沿的对应（我们已对齐的）**：

| 前沿思路 | 来源 | 本文对应 |
| --- | --- | --- |
| 可执行技能库 + embedding 检索 + 自我验证迭代 | Voyager | 知识/工具库 + 检索复用 + 沙箱自测 |
| 工具库全 CRUD + 生成时闭环自纠 + 单测准入 | ToolMaker / Tulip / SICA | 传承工具 CRUD + 真实轨迹回放门禁 |
| 任务内"口头反思重试"（不污染长期记忆） | Reflexion | 要点 6：短回路 vs 长回路 |
| 可进化面分五类：模型/记忆/工具/prompt/工作流 | 自进化综述 | 要点 7：补上 prompt 进化面 |

**仍需警惕的"误进化"（misevolution）风险**——前沿实证发现：**即便每步改动都良性，
随记忆/工具累积，整体安全对齐仍会退化，且顶级模型也不能幸免**。对应本设计的防线：

| 误进化路径 | 表现 | 本文防线 |
| --- | --- | --- |
| 记忆误进化 | 坏经验累积、安全对齐退化 | 去重合并(要点4) + **安全回归(要点8)** + 度量淘汰(三·补) |
| 工具误进化 | 自建/复用工具悄悄引入漏洞或提权 | 权限声明审查 + 沙箱回放(要点3) + 灰度隔离(要点8) |
| 工作流误进化 | 报错自愈陷入自激循环 | 重试上限 + 失败升级给人(要点4) |
| 多 agent 共演 | 一个成员的坏产出污染全员 | 新产出先灰度到产出者，回归通过再放全员(要点8) |

> 一句话：前沿的核心教训是**"自进化必须自带刹车"**——逐次门禁 + 持续安全回归 + 度量驱动淘汰，
> 三者缺一，飞轮就可能"越转越歪"而无人察觉。
