# 进化流程设计（自进化执行闭环）

> 本文记录 HeySure AI 2.0 "知识 + 工具"自进化闭环的设计思路：先复述原始构想，
> 再对照当前代码标出已实现 / 待实现的缺口，最后给出推荐的增强版进化飞轮与落地优先级。
> 角色与知识库现状见 [`角色与知识流转.md`](角色与知识流转.md)。

## 一、核心思路：一条自进化执行闭环

围绕执行体（图中代号「贝」，落地为数字成员 Agent）的一条闭环：

```
执行 → 总结 → 沉淀知识 → 复用 → 进化工具
```

让执行体把"怎么做成的 / 踩了什么坑 / 可复用的工作流"沉淀成**知识**和可演进的
**MCP 工具**，下次自动复用、自动纠错。原始构想（手绘草图）包含四块：

- **传承知识（知识库）**：设备端分类、新建、编辑、删除、读取（列表/详情）、联网下载。
- **角色与数据流**：`知`(知识) / `网`(联网) / `贝`(执行体) / `总`(总结) / `M`(模型) / `工`(工具)
  之间流转；贝产出三类东西——**遇到的报错 / 完整的工作流 / 可简化的工作流**。
- **工作流**：工作流可被「执行」。
- **传承工具（MCP）**：读取(列表/详情)、新建、编辑、删除、**版本(列表/详情/撤回)**；
  并要求 ① 动态加载 MCP ② 主动修改 MCP 代码 ③ 版本回溯。

四条闭环规则（草图右上）：

1. 贝任务执行后对用到的 MCP **主动总结**。
2. 阿（核心管理员/阿尔法）把贝的总结**识别、沉淀成知识库**。
3. 贝**读知识库**来决定 MCP 调用。
4. 贝**读 MCP 库知识自动跑工作流**，并**主动处理报错、再次总结**。

| 构想点 | 现状 | 关键位置 |

| 角色（贝/阿/总） | 有对应角色，但"贝/阿/总"是叙事代号，落地为 `assistant_admin`/`digital_member`/`is_librarian` | `server/api/models/ai_config.py`、`角色与知识流转.md` |

### 待实现 / 半实现（右半边为主，按缺口大小排序）

1. **传承工具（MCP）的增删改** —— 现在只能"读"（列表/describe），不能在产品里
   新建/编辑/删除工具。工具写死在 `server/mcp_runtime/mcp/tools/*.py`，靠热重载生效。**❌**
2. **MCP 版本管理（列表/详情/撤回/回溯）** —— registry 只有自增 `.version` 字段，
   无历史、无回滚。`server/mcp_runtime/mcp/core.py`。**❌**
3. **运行时主动改 MCP 代码** —— 只有"传承思想"的 SKILL.md 能按行编辑，
   MCP 工具本体代码不能在运行时被 AI 改。**❌**
4. **报错自愈 + 读 MCP 知识自动编排工作流** —— 任务出错只把 `AITaskJob.status="error"`
   落库，无自动重试/自愈，也没有工作流编排引擎。**❌**
5. **执行后自动总结 MCP + 总结自动入库** —— 现在需 AI 显式调 `task.complete/inherit`
   传 summary（非执行完自动），且总结对象是任务而非"用到的 MCP 工具本身"；
   入库还要人审。**⚠️ 半实现**
6. **真正的工作流引擎** —— 只有任务 + 定时调度（queued→running→completed，可循环），
   没有 DAG / 分步管线 / "完整工作流 vs 可简化工作流"的表示与执行。**⚠️ 半实现**

**一句话**：左边"知识沉淀与复用"较完整；右边"把 MCP 工具当成可被 AI 增删改、
可版本回溯、可自动编排执行的活资产"基本空白。

## 三、推荐的增强版进化飞轮

相比原始草图，建议增加三样东西：**① 去重/合并闸 ② 沙箱 + 审批门禁
③ 知识→工具的晋升通道 + 成功率反馈降权**。

```
执行(贝)  ──采集调用轨迹(后端结构化)──►  自动初稿总结(模型只做提炼)
   │                                              │
   │◄──── 复用 ────┐                              ▼
   ▼               │                     去重/合并  ──►  draft 知识
读知识库决定调用     │                              │
   │               │                       人/阿审批门禁
   ▼               │                              ▼
报错? ──是──► 重试(有上限) ──仍失败──► 升级给人      active 知识
   │                                              │
   └──否──► 成功                      被高频复用? ──是──►  晋升为 MCP 工具(draft)
                                                   │              │
                                                   │        沙箱自测+审批
                                                   ▼              ▼
                                          带成功率统计       active 工具(可版本回溯)
```

### 设计要点

**1. "知识"和"工具"是两层，且有晋升关系（原图把它们并列了）**
- 知识 = markdown SOP，软的、给模型读的"怎么做"。
- 工具 = 可执行 MCP 代码，硬的、确定性的。
- 进化的关键：某条 SOP 被反复使用 → **晋升**为一个 MCP 工具（把"每次让模型照着做"
  固化成"一次调用必成功"）。应显式加一条 **知识 → 工具 晋升通道**，而非两条独立 CRUD。

**2. AI 自动改/建 MCP 代码 = 最大风险点，必须"隔离 + 门禁"**
复用现成模式 `librarian.propose → 用户审批 → active`，让 MCP 工具的新建/改代码走同一套：
- AI 只能提交到 **draft/灰度** 状态，不能直接热加载进生产 registry；
- 新工具先在 **沙箱**（子进程/受限权限）跑通"自测样例"才允许晋升；
- 人审（或更高角色 AI 审）通过 → 才 active。
- 这正好把草图右上②（阿识别）与右下②（改代码）接起来：**阿/管理员当审批闸**。

**3. 闭环要防"知识污染"和"自激震荡"**
- 总结入库前做 **去重/合并**（同 trigger 命中已有条目则 merge，不新建）；
- 每条知识/工具带 **使用统计 + 成功率**，低成功率自动降权或触发复审；
- 报错自愈要有 **重试上限 + 失败升级给人**，不能无限自循环。

**4. "自动总结"不要做成"执行完调一次大模型凭记忆复述"**
- 总结**原料**由后端结构化采集（用了哪些 MCP、入参/出参、报错栈、耗时、最终成败），
  模型只负责"提炼成 SOP"。
- 现有 `task.inherit` 已在自动抓 unfinished/artifacts/token，把这套扩展到
  "MCP 调用轨迹"即可，更省钱也更准。

## 四、落地优先级

1. **传承工具 CRUD + 版本管理/回溯** —— 缺口最大、且不碰危险的"AI 自动改代码"，
   纯人工管理 MCP 也立刻有价值。先做。
2. **MCP 调用轨迹采集 + 自动初稿总结** —— 复用现有 `task.inherit` 钩子。
3. **AI 自动建/改工具（沙箱 + 审批）+ 报错自愈** —— 风险最高，放最后，且必须带门禁。
