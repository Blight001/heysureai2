# HeySure AI 2.0 — 升级优化计划

**制定时间：** 2026-05-21  
**参考基准：** Claude Agent SDK / Claude Code 多智能体架构模式

---

## 背景

当前项目已完成三层架构（web / server / agent）的基础搭建，多 AI 角色管理、任务调度、MCP 工具调用均已打通。对比 Claude 官方多智能体最佳实践，存在以下几类系统性差距：

- MCP 工具调用依赖文本解析，稳定性差
- 没有 Prompt Caching，token 成本随上下文线性增长
- AI 间只能通过数据库间接通信，无法真正并行协作
- 桌面 agent 端只是骨架，未接入当前任务体系
- 上下文传承机制粗糙，知识损耗严重

## OPT-01：切换原生 tool_use API

### 问题

`chat_worker.py` 当前通过正则解析 AI 文本输出来提取 MCP 工具调用：

```python
# 当前方式 — 脆弱
pattern = re.compile(r"<mcp[-_]call>\s*([\s\S]*?)\s*</mcp[-_]call>", re.IGNORECASE)
```

这意味着：
- AI 格式稍有偏差（多空格、换行位置不对）就解析失败
- AI 需要在 system prompt 里被明确告知"必须输出 `<mcp_call>` 标签"
- 不能利用模型原生的工具调用能力（更准确、更快）

### 目标

改为使用 OpenAI-compatible 原生 `tools` 参数，接收 `tool_calls` 内容块：

```python
# 目标方式 — 原生 tool_use
payload = {
    "model": model,
    "messages": convo,
    "tools": build_tool_definitions(effective_tool_allowlist),
    "tool_choice": "auto",
    "stream": True,
}
# 响应中 finish_reason == "tool_calls" 时，直接读 tool_calls 字段
```

### 影响文件

- `server/api/routers/chat_worker.py` — 主循环重写
- `server/api/routers/chat_prompt_utils.py` — 删除文本解析逻辑，改为 tool_calls 解析
- `server/api/mcp_registry_setup.py` — 需要输出 OpenAI tools 格式的 JSON Schema
- `server/api/routers/chat_base.py` — system prompt 中移除"格式要求"提示

### 实施步骤

1. 在 `mcp_registry_setup.py` 给每个工具补充完整的 JSON Schema，生成 `build_tool_definitions()` 函数
2. 修改 `chat_worker.py` 的请求 payload，加入 `tools` 参数
3. 修改流式响应解析：检测 `finish_reason == "tool_calls"`，读取 `tool_calls[0]`
4. 修改 `convo` 追加逻辑：按 OpenAI 格式追加 `tool` role 消息
5. 清理 system prompt 中不再需要的 MCP 格式说明
6. 兼容处理：非 tool_calls 模型（如某些 DeepSeek 版本）保留文本解析作为降级

### 验收标准

- AI 不再需要输出 `<mcp_call>` 标签
- 工具调用成功率 > 99%（当前约 90%）
- system prompt 字数减少 200+ token

## OPT-02：接入 Prompt Caching

### 问题

每轮对话将完整的 system prompt + 全部历史消息重新发送给模型，token 成本随对话轮次线性增长。一个执行 20 轮的任务，system prompt 会被计费 20 次。

### 目标

对 Anthropic Claude 系列模型使用 `cache_control` 标记，对 OpenAI 兼容 API 使用 prefix caching（自动，无需特殊处理）：

```python
# Anthropic SDK 格式
{
    "role": "system",
    "content": [
        {
            "type": "text",
            "text": system_prompt,
            "cache_control": {"type": "ephemeral"}  # 缓存 system prompt
        }
    ]
}
```

### 适用范围

- 长任务 session（generation > 1 的任务对话）
- system prompt 超过 1024 token 的 AI 配置
- 使用 Anthropic Claude 模型的配置（检测 `base_url` 是否为 Anthropic API）

### 影响文件

- `server/api/routers/chat_worker.py` — 构建 convo 时按模型类型加 cache_control
- `server/api/routers/chat_runtime_helpers.py` — 增加 `_detect_provider(base_url)` 函数

### 实施步骤

1. 实现 `_detect_provider(base_url) -> "anthropic" | "openai_compat"` 工具函数
2. 当 provider 为 anthropic 时，将 system prompt 包装为带 `cache_control` 的 content 数组
3. 将历史中最后 N 条之前的消息也加 cache_control（prefix caching）
4. 在 token 统计中区分 `cache_read_input_tokens` 和 `prompt_tokens`，写入数据库备查
5. 在前端 AI 卡片展示缓存命中情况（可选）

### 验收标准

- 长任务（> 5 轮）的 prompt token 成本降低 ≥ 60%
- 不影响非 Anthropic 模型的正常工作

## OPT-03：桌面 Agent 真正对接

### 问题

`agent/` 目录只有约 70 行骨架代码，监听的 `flow:run` 事件与 server 当前任务体系（`AITaskJob`、MCP 工具）完全脱节。三层架构名存实亡。

### 目标

Desktop agent 能够：
1. 注册到 server，作为可调度的执行节点
2. 接收任务，在本地执行（访问本地文件系统、运行命令、调用本地 MCP）
3. 执行结果回传 server，写入任务会话历史

### 架构设计

```
Server (调度中心)
  ↓ Socket.IO: task:dispatch
Desktop Agent (本地执行节点)
  ├── 接收任务指令
  ├── 调用本地 LLM API（或转发到 server）
  ├── 执行本地 MCP 工具（文件系统、shell、浏览器）
  └── 回传结果: task:result → server 写入会话
```

### 新增 Socket 事件协议

| 事件 | 方向 | 说明 |
|------|------|------|
| `agent:register` | agent → server | 注册时携带 capabilities（支持哪些工具） |
| `task:dispatch` | server → agent | 下发任务（含 job_id、instruction、allowed_tools） |
| `task:progress` | agent → server | 实时进度推送 |
| `task:result` | agent → server | 任务完成，回传结果摘要 |
| `task:error` | agent → server | 任务失败 |

### 影响文件

- `agent/src/index.ts` — 实现新协议，替换 flow:run
- `agent/src/executor.ts` — 重写为 TaskExecutor，对接本地 MCP 工具
- `agent/src/tools/` — 新增本地工具模块（filesystem、shell、git）
- `server/api/socket_events.py` — 增加 task:result / task:progress 处理
- `server/api/routers/execute.py` — 增加 task:dispatch 下发接口

### 实施步骤

1. 定义 agent 注册协议（capabilities 字段）
2. 重写 `agent/src/index.ts`，实现新事件处理
3. 实现 `TaskExecutor`：接收 job，调用本地工具，结果回传
4. server 端增加 `task:result` 处理：将结果写入 `ChatMessage`
5. 前端展示 agent 来源标识（区分 server 执行 vs agent 执行）

### 验收标准

- Desktop agent 启动后出现在前端 admin 面板的 agent 列表
- 可以从前端向 desktop agent 下发一个"列出本地文件"任务并看到结果

## OPT-04：并行 subagent 执行

### 问题

当前调度器对每个 AI 同一时间只允许一个 running 任务，任务之间串行等待，无法发挥多 AI 并行的优势。主脑也无法在一次任务中同时驱动多个下属 AI 并发执行子任务。

### 目标

支持 orchestrator-worker 模式：
- 主脑 AI 可以通过 MCP 工具 `task.create` 向多个 digital_member 同时派发子任务
- 主脑通过 `task.list` 查看子任务状态，结合普通对话/通知机制汇总结果继续

### 实施步骤

1. 调度器允许同一 AI 同时运行多个任务（去掉"running 时不启动新任务"限制，改为并发上限控制）
2. 保持任务工具精简，不再提供阻塞等待类 MCP 工具
3. 在 system prompt 中说明 orchestrator 工作模式

### 验收标准

- 主脑可以一次派发 3 个子任务，同时执行，最终汇总

### 实现说明（已完成）

实测发现调度器 `process_task_scheduler` 是**按 AI 配置逐个处理**的：不同
digital_member 各自独立判断是否启动任务，因此**跨成员的并行执行本就成立**。
当前任务 MCP 已移除阻塞等待类工具，避免任务工具集膨胀和长时间占用调用链。
保留 orchestrator-worker 模式的非阻塞编排：

1. **管理者扇出**：`_resolve_task_runtime_owner` 放开限制——`digital_member_role` 为
   `manager` 的成员可通过 `target_ai_config_id` 为其它成员创建子任务，从而由调度器
   并行拉起。
2. **状态查看**：管理者使用 `task.list` / `task.get_current` 查看子任务状态；
   需要汇总时由上层对话流程或消息工具继续推进。

> 出于稳定性考虑，未拆除"单个 AI 配置同一时刻仅一个 running 任务"的串行约束
> （该约束与代际/监督/抢占逻辑强耦合），并行能力通过**跨成员扇出**实现。

## OPT-05：结构化 Memory 替代 summary 传承

### 问题

当前任务传承（`task.inherit`）只是把上一代最后一条助手消息截取 1200 字拼进下一代 prompt，信息损耗大，且没有结构化检索能力。

### 目标

引入持久化知识库，AI 在任务执行中可以：
- 写入知识条目（`memory.write`）
- 检索相关知识（`memory.search`）
- 传承时附带结构化摘要（完成事项、关键结论、待处理风险）

### 数据结构

```json
{
  "memory_id": "mem_xxx",
  "ai_config_id": 1,
  "job_id": "job_xxx",
  "generation": 2,
  "tags": ["文件管理", "已完成"],
  "content": "2026-05-21 完成了文件目录整理，共处理 42 个文件",
  "created_at": 1234567890
}
```

### 新增 MCP 工具

| 工具 | 说明 |
|------|------|
| `memory.write` | 写入知识条目（支持 tags） |
| `memory.search` | 按关键词/tag 检索相关记忆 |
| `memory.list` | 列出最近 N 条记忆 |

### 影响文件

- `server/api/models.py` — 新增 `AIMemory` 数据模型
- `server/api/mcp_registry_setup.py` — 注册 memory.* 工具
- `server/api/mcp_memory_tools.py` — 新增，实现 memory 工具逻辑
- `server/api/routers/chat_scheduler.py` — 传承时自动附加最近记忆

### 验收标准

- AI 执行任务时可调用 `memory.write` 写入，下一代任务启动时自动看到最近记忆
- 跨任务的知识可被检索到

## OPT-06：Human-in-the-loop 标准化

### 问题

AI 遇到不确定情况时，只能用自然语言输出"请人工确认"，没有标准化的暂停-等待-继续机制。任务会 timeout 或 error，而不是优雅地等待人工响应。

### 目标

新增 `human.ask` MCP 工具，AI 调用后任务暂停，前端弹出确认对话框，用户回答后任务自动继续。

### 新增 MCP 工具

```
human.ask(question: str, options?: list[str]) -> {answer: str}
  — 暂停任务，向用户提问，返回用户的回答
```

### 实施步骤

1. 新增 `human_ask` 任务状态（介于 running 和 paused 之间）
2. `human.ask` 调用后向前端推送 Socket 事件，前端弹出确认框
3. 用户回答后前端 POST `/api/chat/run/{run_id}/resume`，携带回答内容
4. worker 恢复执行，将用户回答作为 tool result 追加进 convo

### 验收标准

- AI 调用 `human.ask` 后任务暂停，前端出现确认框
- 用户点击确认后任务继续，AI 能看到用户的回答

## OPT-07：threading → asyncio 并发重构

### 问题

当前每个 AI run 启动一个 `threading.Thread`，多任务并发时：
- 数据库读写无显式锁，可能出现竞态
- `asyncio.run()` 在 thread 内嵌套调用，有潜在死锁风险
- 线程数随并发任务线性增长，无上限控制

### 目标

将 worker 改为 `asyncio` 协程，统一在事件循环中调度：

```python
# 目标架构
async def _run_worker_async(...):
    async for chunk in stream_response(...):
        ...
    result = await registry.call_async(tool, ...)
```

### 影响文件

- `server/api/routers/chat_worker.py` — 全量改为 async
- `server/api/routers/chat_scheduler.py` — 改为 asyncio task 管理
- `server/api/routers/chat_base.py` — 替换 `_RUN_THREADS` 为 task 字典
- `server/api/mcp.py` — registry.call 改为 async

### 注意

此项改动量最大，影响范围广，建议在 OPT-01 ~ OPT-05 稳定后再做，避免同时引入多个变量。

### 验收标准

- 移除所有 `threading.Thread` 和 `asyncio.run()` 嵌套调用
- 压力测试：10 个 AI 同时执行任务，无死锁、无数据库竞态
