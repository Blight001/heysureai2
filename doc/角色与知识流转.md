# 角色与知识流转说明

> 本文回答两个常被问到的问题：
> 1. 这套"数字社会"里到底有哪几个 AI 角色，各自管什么、prompt 在哪、代码在哪？
> 2. 图书管理员的"记忆 / 检索 / 派任务"链路是怎么闭环的？
>
> 读 prompt 时容易被"英灵殿 / 传宗接代 / 文明守护者"这类叙事词带偏，本文把**叙事层**和**指令层**拆开，给一张"魔幻词 → 技术内核"对照表。

---

## 一、角色清单

| 角色 | 代号 | 运行位置 | 职责（去叙事化） | prompt 文件 | 关键代码 |
| --- | --- | --- | --- | --- | --- |
| 核心管理员 / Archivist | 主脑·阿尔法 | Server | **治理中枢**：响应用户指令、拆分项目为 AI 岗位、创建/调度 Agent、写使命书、归档知识 | [`doc/prompt/数字社会核心管理员prompt.md`](prompt/数字社会核心管理员prompt.md)（满配）<br>[`doc/prompt/核心管理员prompt.md`](prompt/核心管理员prompt.md)（精简） | `server/api/chat_runtime/` 编排 |
| 辅助管理员 | — | Server | **只读观测助手**：查进程/资源/服务健康、汇总 AI 运行态、给排查建议。**不做治理与调度** | [`doc/prompt/辅助管理员prompt.md`](prompt/辅助管理员prompt.md) | — |
| 图书管理员 / Librarian | — | Server | **知识门面**：收沉淀申请、按需检索流程知识、派任务前自动注入简报。**不派任务本身** | （无独立 prompt，靠 `is_librarian` 标记 + MCP 工具） | `server/api/services/librarian_service.py`<br>`server/mcp_runtime/mcp/tools/librarian.py` |

| 桌面助手 | 德尔塔 | Windows 物理机 | **桌面执行器**：文件/命令/截图/键鼠/窗口/进程/剪贴板操作 | [`doc/prompt/桌面助手prompt.md`](prompt/桌面助手prompt.md) | `agent/windows/src/tools/` |
| 浏览器助手 | 贝塔 | 浏览器扩展 | **浏览器执行器**：导航/表单/截图/抓取/自动化测试 | [`doc/prompt/浏览器助手prompt.md`](prompt/浏览器助手prompt.md) | `agent/extension/src/` |
| （共享）工具调用协议 | — | 所有 AI | 规定 `<mcp-call>` XML 块的输出格式 | [`doc/prompt/MCP工具调用prompt.md`](prompt/MCP工具调用prompt.md) | `server/api/chat_runtime/`（MCP 解析） |

**层级关系**：

```
用户
  │  指令（查状态 / 创建项目 / 停止项目）
  ▼
核心管理员（阿尔法）── 治理 / 拆岗位 / 派项目 / 归档
  │
  ├──（派任务那一刻，系统自动调用）──► 图书管理员 ── 注入相关经验简报
  │
  └── 调度执行成员
        ├── 桌面助手（德尔塔，Windows）
        └── 浏览器助手（贝塔，扩展）

辅助管理员 ── 旁路：只读观测 server / AI 运行态，不在主链路上
```

> ⚠️ 两个最常见的理解偏差：
> 1. **辅助管理员不"管辖 AI 组"**——它只是只读观测助手；真正治理/派项目的是核心管理员。
> 2. **图书管理员不"安排任务"**——它是知识层；派任务时被自动调用、把经验注入进去，派活动作本身由 task system + 核心管理员完成。

### 1.1 架构变更：知识与进化迁出为工坊 agent + 管理员角色扁平化

两项结构性调整（详见 [`server/workshop/README.md`](../server/workshop/README.md)）：

1. **知识与进化工坊**：`librarian.*` / `evolution.*` 不再是服务端内置 MCP 工具，
   改由集中存放在 `server/workshop/` 的"工坊"提供——服务端内置的虚拟端侧，
   **每个账号自动上线**（作坊面板与社会显示中自动出现，无需用户运行程序）。
   - AI 必须**绑定工坊**才能看到/调用这些工具（`WorkshopAiBinding`，**1:1**：
     工坊只能绑定一个 AI 数字成员，新绑定替换旧绑定）：AI 配置弹窗勾选，
     或在游戏世界把成员拖到工坊建筑上；
   - 工具执行经调度层进程内调用 `workshop.engine.execute_tool`，数据真相源
     （DB + KnowledgeBase 文件）始终在服务端，每次调用复核归属/白名单/角色/绑定；
   - **只改 `server/workshop/` 目录即可控制知识与进化方向**：`direction.md`（方向指引，
     随工具结果实时注入给 AI，保存即生效）、`policy.py`（入参/结果钩子）、
     `tools.py`（工具描述）。
2. **角色扁平化**：不再支持新建/提升辅助管理员；注册时系统仍默认创建一个
   （`ensure_default_configs`），其余 AI 一律按数字成员对待。`evolution.review`
   的最低角色从 assistant_admin 降为 manager（数字成员·管理者即可审批）。

## 二、图书管理员：记忆与知识流转闭环

图书管理员 = 一个被 `AssistantAIConfig.is_librarian=true` 标记的特殊 AI（每用户最多一个，`librarian_service.py:231`）。它不是聊天机器人，而是 **KnowledgeBase 的检索门面**，设计参考了 Claude Code Skills 的"渐进披露"（先看标题、按需读全文）。

### 2.1 知识长什么样（"习惯性记忆 + 典型错误记忆"）

每条知识落盘为一份 markdown：`<workspace>/KnowledgeBase/topics/<slug>.md`，DB 里 `KnowledgeEntry` 建索引。结构固定三段（`_render_procedure_md` `librarian_service.py:323`）：

| 段落 | 含义 | 类比 |
| --- | --- | --- |
| `## 场景 / 触发条件`（scenario）+ `triggers` 关键词 | 什么时候该用这条知识 | 索引 / 触发器 |
| `## 操作步骤`（steps） | 习惯性、可复用的做法 | "习惯性记忆" |
| `## 注意事项 / 已知坑`（gotchas） | 典型错误、风险、踩坑点 | "典型错误记忆" |

### 2.2 完整闭环

```
① 沉淀申请 propose（任何数字成员可调）
     成员把"这次怎么做成的 + 踩了什么坑"提交
     → status=pending，落盘 topics/<slug>.md + KnowledgeEntry 一行
          │
② 用户审批 approve / reject（前端 ProposalReviewModal.vue）
     approve → status=active（可被检索）
     reject  → status=rejected（文件留痕，索引剔除）
          │  AI 不能自己往长期记忆写脏数据，必须人审
          ▼
③ 派任务前自动简报 brief（chat_scheduler.py:153）
     用任务标题匹配 active 条目 → 命中则注入
     "[图书管理员预先简报] …如需细节调 librarian.read(memory_id)"
          │
④ 成员主动检索 consult（chat_scheduler 之外，AI 随时可调）
     两阶段关键词打分检索（librarian_service.py:1491）
          │
⑤ 渐进披露 list_topics → read
     list_topics 只返标题/触发词/摘要；按需 read 拉全文
          │
⑥ 归档 archive（仅图书管理员）
     过时知识移出默认检索
```

### 2.3 MCP 工具与权限

工具由知识与进化工坊（`server/workshop/`，每账号自动上线）提供，**调用前提是
该 AI 已绑定工坊**；handler 实现仍在 `server/mcp_runtime/mcp/tools/librarian.py`
（经 `workshop.engine.execute_tool` 进程内执行）：

| 工具 | 谁能调 | 作用 |
| --- | --- | --- |
| `librarian.propose` | 任何数字成员 | 提交沉淀申请（→ pending） |
| `librarian.consult` | 任何数字成员 | 按 query 检索，返回最多 k 条完整步骤 |
| `librarian.list_topics` | 任何数字成员 | 只列标题/触发词/摘要（渐进披露） |
| `librarian.read` | 任何数字成员 | 按 memory_id 读全文 |
| `librarian.archive` | **仅图书管理员** | 归档条目 |

> 审批（approve/reject）不是 MCP 工具，而是走 REST + 前端（`librarian_routes.py` / `web/src/api/librarian.ts` / `ProposalReviewModal.vue`），因为它需要人工介入。

## 三、prompt 的"叙事层 vs 指令层"

核心管理员 prompt（尤其满配版 `数字社会核心管理员prompt.md`）用了大量拟人/史诗化词汇。这些词对**人类理解产品理念**有叙事价值，但写进**喂给模型的 system prompt** 会鼓励角色扮演与抒情——这恰好和同项目其它 prompt 反复强调的"不要角色扮演、不要自言自语、每句话要有工具结果支撑"相矛盾（见 `辅助管理员prompt.md` / `浏览器助手prompt.md` 的 `[内部执行规则]`）。

阅读时按下表"解码"即可：

| 叙事词（prompt 里） | 技术内核 | 对应实现 |
| --- | --- | --- |
| 数字社会 / 文明守护者 | 一个用户名下的 AI 集合 + 治理逻辑 | `AssistantAIConfig` 表 |
| 进化竞技场 EvolutionArena | 活跃 Agent 的档案目录 + `registry.json` 索引 | workspace 文件 + DB |
| 英灵殿 Valhalla / 遗言 / 逝者 | 已退场 Agent 的归档 + `memorial_registry.json` | 归档目录 |
| 传宗接代 | token 将耗尽时的上下文交接（继任 Agent 读归档继承上下文） | task.inherit / 传承简报 |
| 主脑·阿尔法 / 贝塔 / 德尔塔 | 给各角色起的代号 | prompt 内代号 |
| 见证诞生与退场 / 沉淀经验 | 写日志 + 知识压缩归档 | 图书管理员 + journal |

**建议**：叙事（Valhalla、传承理念）留在 README / 本文讲故事；真正喂模型的 system prompt 用中性术语（active agents / archived agents / context handoff / knowledge base），并保留行为约束。这样不丢产品调性，又能减少模型抒情和幻觉。

### 3.1 核心管理员 prompt 的机制已迁移（重要）

比"措辞魔幻"更需要注意的是：满配版 `数字社会核心管理员prompt.md` 描述的**机制**已经和代码脱节。它假设一个"管理员亲手用文件系统建目录、写 `registry.json` / `last_words.md`"的世界，但代码早已迁移：

| prompt 旧描述 | 代码现状 |
| --- | --- |
| 进化竞技场 EvolutionArena = 活跃 AI 档案目录 + `registry.json` | 该表在 `migrations.py:396` 作为死表删除；活跃成员就是 `AssistantAIConfig` 数据库表，用 `admin.get_overview` / `admin.list_agents` 查 |
| 英灵殿 Valhalla = 管理员建文件夹、收 `last_words.md` | `valhalla_service.py:4`：数据全存 `ValhallaEntry` 数据库，**不再落任何文件**，且**后端钩子自动写入、AI 无感** |
| 管理员手工"接收并规范化遗言" | 成员调 `task.inherit(summary)` → 后端自动归档 → `chat_scheduler` 自动注入下一代。管理员不碰 |
| Archivist 自由读写知识库 | 知识库由**图书管理员**维护，走 `librarian.propose → 用户审批 → active` 闭环 |

换言之，原 Archivist 的"知识 + 英灵殿"职责已被**拆走**（知识→图书管理员，交接→后端自动），核心管理员剩下的真实职责是**协调调度**：响应用户、拆项目、`task.create` 派活给现有成员、`message.send_to_ai` 协作、`evolution.review` 评审。注意**没有"创建 Agent/项目"的 MCP 工具**——新建成员是用户在控制台做的事。

> 据此重写的中性 + 对齐现实版见 [`数字社会核心管理员prompt.中性版.md`](prompt/数字社会核心管理员prompt.中性版.md)，与原文 [`数字社会核心管理员prompt.md`](prompt/数字社会核心管理员prompt.md) 并存以便对比。落地前请人工确认线上实际使用的是哪一份（运行时 prompt 实际存在 DB 里，`doc/prompt/*.md` 为参考/种子文档）。
