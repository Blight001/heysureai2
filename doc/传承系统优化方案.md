## 0. 背景与参考

### 0.1 当前实现的硬伤（已通过代码确认）

| 问题 | 关键代码位置 |
|------|--------------|
| `Valhalla / KnowledgeBase / EvolutionArena` 三个文件夹只是 `mkdir`，从未写入 | `server/api/mcp_core.py:72`、`server/api/routers/auth.py:27` |

| 没有 Archivist / Librarian 角色实体，主脑·阿尔法跟普通 manager 无差别 | `server/api/ai_service.py:120-134` |

### 0.2 参考的 Anthropic / Claude Code 公开方案

1. **Claude Memory Tool（2026-04 公开测试版）**
   把记忆存成"文件系统目录"，模型可 CRUD 这些文件；Netflix / Rakuten 等早期客户报出"首轮错误率 -97%、文档核验速度 +30%"。文件式存储的好处是：**人类可读、可导出、可手动编辑、可 diff**。我们的方案沿用这个范式。

2. **MCP Knowledge Graph Memory Server（modelcontextprotocol/servers/src/memory）**
   官方参考实现把记忆抽象成三种原语：
   - **Entities**（实体）
   - **Relations**（实体间关系，主动语态）
   - **Observations**（实体上的离散观察）

   存储就是一个 `memory.jsonl`。这个最小化原语集对我们沉淀"操作步骤"刚好够用——可以把"流程"看作一组带顺序的 Observations，挂在 `Procedure` 实体上。

3. **Claude Code 的 CLAUDE.md + Auto Memory + Skills 三层架构**
   - **CLAUDE.md** = 总是加载的稳定指令（小、聚焦、模块化）
   - **Auto Memory** = Claude 自己根据"用户纠正/偏好"沉淀的笔记
   - **Skills（progressive disclosure）** = 启动时只加载 skill 名称+一行简介，**触发时才加载完整内容**。这是关键：让 AI 有"几十条经验可用"而不撑爆上下文。

4. **CLAUDE.md 反例**：官方建议 > 200 行就开始"互相矛盾、难维护"。这告诉我们：**传承知识库必须按主题拆分文件 + 渐进披露**，不要把所有知识塞进一个大文档喂给每一代。

5. **Context Rot 现象**：上下文不超过技术上限，但塞太满时模型质量会下降。所以"派发任务前简报"必须**精简、相关、有出处**。

## 1. 设计目标

| 编号 | 目标 | 衡量标准 |
|------|------|----------|
| G1 | **英灵殿要真有英灵** | 每一代任务结束 / 传承时落一份可读 markdown；前端能按"任务 → 代际"翻阅 |
| G2 | **传承知识库要能跨任务、跨 AI、跨代复用** | 同一类问题第二次出现时，AI 不用从头摸索；用户不用重复教 |
| G3 | **图书管理员是"渐进披露"的入口**，不是大喇叭 | 单次 consult 注入的 token < 1500；命中率 > 60% |
| G4 | **用户对沉淀有最终决定权** | 所有写入知识库的条目都必须显式 `pending → approved` |
| G5 | **不破坏现有任务/代际/run 链路** | 现有 `task.inherit` 旧调用照常成功；新能力以"叠加"方式启用 |

## 2. 总体架构

```
                       ┌────────────────────────────────────────────┐
                       │              用户 (上帝视角)                │
                       │  - 确认/驳回知识沉淀建议                    │
                       │  - 翻阅英灵殿 / 知识库                       │
                       └─────────┬────────────────────┬─────────────┘
                                 │ approve/reject     │ browse
                                 ▼                    ▼
┌─────────────────────────────────────────────────────────────────────┐
│                      图书管理员 Librarian (AI Role)                  │
│   - 复用现有 digital_member + 标志位 is_librarian=true              │
│   - 独占 librarian.* MCP 工具集                                      │
│   - 听任务调度的钩子：在 _start_task_run 之前做"预先简报"            │
└──┬─────────────────────────────┬─────────────────────────────┬─────┘
   │ 写                          │ 读/检索                      │ 读
   ▼                             ▼                             ▼
┌──────────────────┐   ┌──────────────────────────┐   ┌──────────────────┐
│ 英灵殿  Valhalla │   │ 传承知识库 KnowledgeBase  │   │   工作日志(现有)  │
│ (event log)      │   │ (procedural memory)       │   │  ChatMessage     │
│                  │   │                           │   │  AITaskJob       │
│ 文件 + 索引表    │   │ 文件 + 索引表 + 向量索引   │   │  ChatRun         │
└──────────────────┘   └──────────────────────────┘   └──────────────────┘
        ▲                          ▲
        │ task.inherit 触发         │ AI 工作中
        │ task 完成触发              │ librarian.consult
        │                          │
┌────────┴──────────────────────────┴─────────────────────────────────┐
│              AI 员工 (digital_member, 非 librarian)                  │
│   - 主动调 librarian.consult(query) 求助                             │
│   - 主动调 librarian.propose(title, body) 申请沉淀                    │
│   - 任务结束自动落英灵殿（由后端 hook 完成，AI 无感）                  │
└──────────────────────────────────────────────────────────────────────┘
```

**三句话总结：**
- **英灵殿** = 事件追加日志（append-only），记录"每一代发生了什么"；
- **传承知识库** = 程序性记忆库，记录"以后遇到 X 怎么做"；
- **图书管理员** = 一个特殊的 digital_member，是知识库的唯一写入者和首要检索者。

## 3. 三大组件详细设计

### 3.1 英灵殿 Valhalla（事件追加日志）

#### 3.1.1 写入时机（共 3 个，由后端 hook 自动触发，不依赖 AI 自觉）

| 触发点 | 当前代码位置 | 写入内容 |
|--------|--------------|----------|
| `task.inherit` 成功后 | `server/api/routers/chat_worker.py:885-921` | 本代的 `last_words.md`（summary + token + 未完成事项 + 产出文件） |
| `task.complete` 成功后 | `server/api/routers/chat_worker.py`（搜 `task.complete` 分支） | 本代的 `final_words.md`（任务结论） |
| run 进入 `stopped/error` 终态 | `server/api/routers/chat_scheduler.py` 调度器 paused 分支 | 本代的 `aborted.md`（中断原因 + 当时上下文） |

> 关键：**AI 不需要"知道"英灵殿存在**。这是后端的事件持久化，类似审计日志。AI 只需正常调 `task.inherit / task.complete`。

#### 3.1.2 文件结构

```
<workspace_root>/Valhalla/
├── memorial_registry.json          # 索引：所有逝去代际的清单（用于前端 list）
├── <job_id>/
│   ├── job_meta.json               # 任务标题、创建者、创建时间、最终状态
│   ├── g1/
│   │   ├── last_words.md           # 传承时落
│   │   ├── final_words.md          # 完成时落（仅最后一代有）
│   │   ├── unfinished.json         # 结构化的未完成事项（供下一代直接读取）
│   │   ├── artifacts.json          # 本代产出/修改的文件清单（来自 workspace.write_file 调用记录）
│   │   └── token_report.json       # token 消耗明细
│   ├── g2/
│   └── ...
```

### 3.2 传承知识库 KnowledgeBase（程序性记忆）

#### 3.2.1 一条"知识"是什么

借鉴 MCP Knowledge Graph 的"Observation 挂在 Entity 上"思路，但简化为**单一概念：Procedure（流程）**。每条 Procedure 一个 markdown 文件 + 一行索引表记录。

```yaml
# Procedure 数据契约
title: "淘宝商品搜索时遇到登录态丢失的处理流程"
triggers: ["淘宝", "登录", "购物", "taobao", "login required"]   # Skills 风格触发关键词
scope: "global" | "ai:<config_id>" | "project:<project_id>"      # 适用范围
scenario: |
  当通过 workspace.run_command 调浏览器自动化访问 taobao.com
  搜索商品时，若返回 HTML 包含 "登录" 字样或 302 跳转 login.taobao.com，
  说明会话已失效。
steps:
  - "通过 human.ask 提示用户："请打开浏览器扫码登录淘宝，登录后回复 ok"
  - "用户回复 ok 后，再次调 workspace.run_command 重试原命令"
  - "如果仍失败，调 human.ask 询问用户是否需要切换账号"
gotchas:
  - "不要尝试自动填密码，淘宝有反爬"
  - "登录态以 cookies 文件持久化在 ./taobao_cookies.json"
source:
  job_id: "job_aabb1234"
  generation: 2
  ai_config_id: 12
  conversation_msg_id: 9876
confidence: 1.0          # user_confirmed=1.0, auto=0.5, disputed=0.1
status: "active"         # pending / active / archived / rejected
created_at: 1748000000
updated_at: 1748000000
```

#### 3.2.2 文件结构

```
<workspace_root>/KnowledgeBase/
├── index.json                          # 全部 Procedure 的轻量索引（title + triggers + scope + status + confidence）
├── topics/
│   ├── taobao-login.md                 # 一条 Procedure 一个文件
│   ├── git-conflict-resolve.md
│   └── ...
├── archives/
│   └── 2026-05/                        # 按月归档被废弃/合并的条目
└── consolidations/                     # 由多条条目合并出的"主题摘要"
    └── browser-automation.md
```

**为什么 markdown + json index 而不是纯数据库？**
- 人类可直接读、可手动编辑、可 git diff 评审；
- 跨用户、跨服务器导出迁移容易；
- 数据库表只做检索加速，不存"真相"——和 Claude Memory Tool 的设计哲学一致。

#### 3.2.3 数据库索引表（用于检索）

```python
class KnowledgeEntry(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    memory_id: str = Field(unique=True)         # mem_xxx
    user_id: int = Field(index=True)
    title: str
    triggers: str                                # 逗号分隔
    scope: str = Field(default="global")
    scope_target: Optional[str] = None           # ai_config_id / project_id
    file_path: str                               # 相对 workspace 路径
    summary: str = Field(default="")             # 1-2 句的检索摘要
    embedding: Optional[bytes] = None            # 向量（float32 数组打包）
    status: str = Field(default="pending")
    confidence: float = Field(default=0.5)
    use_count: int = Field(default=0)            # 被 consult 命中次数
    last_used_at: Optional[float] = None
    source_job_id: Optional[str] = None
    source_generation: Optional[int] = None
    created_at: float
    updated_at: float
```

**注意**：向量字段可选。**P0 阶段先用 SQLite FTS5 全文检索**（关键词 + 触发词命中），等数据量上来再补向量。模型支持 OpenAI 兼容 embedding API（沿用 AI 配置的 `base_url`，免引入新依赖）。

#### 3.2.4 与现有 `Memory` 表的关系

`server/api/models.py` 已经有 `Memory` 模型（来自 OPT-05 的脚手架）和 `mcp_memory_tools.py`。**复用而非重建**：
- `KnowledgeEntry` 视作 `Memory` 的 superset；
- 迁移路径：把现有 `Memory` 表加上 `triggers / file_path / status / confidence / use_count` 字段；
- `memory.*` MCP 工具保留向后兼容，新增 `librarian.*` 在其之上做"带审批 + 带文件落盘 + 带检索"的封装。

### 3.3 图书管理员 Librarian（检索/沉淀 Agent）

#### 3.3.1 角色实体化

不新建一类 `ai_role`，而是给现有 `digital_member` 加一个布尔标志：

```python
# server/api/models.py 中 AssistantAIConfig
is_librarian: bool = Field(default=False)
```

- 默认种子（`server/api/ai_service.py:120` 的"主脑·阿尔法"）改为 `is_librarian=True`；
- 全用户最多 1 个 librarian（约束：相同 user_id 下 `is_librarian=true` 至多 1 条）；
- librarian 自动获得 `librarian.*` 工具白名单；
- librarian 的 prompt 由系统拼接固定模板（见 §3.3.3）。

#### 3.3.2 librarian 的三种工作模式

| 模式 | 触发方 | 描述 |
|------|--------|------|
| **A. 预先简报（pre-task brief）** | 后端在 `_start_task_run` 之前主动调用 | 在开任务前，librarian 先看任务标题/指令，从 KB 检索 top-K，把"建议参考的流程"压成 1-2 段塞进新代的 system message |
| **B. 在线咨询（on-demand consult）** | AI 员工运行中调 `librarian.consult` | 员工卡住时主动问；librarian 检索后返回精简答复 |
| **C. 沉淀审批（curate proposals）** | 用户在前端点击"确认沉淀" | librarian 把 pending 条目转 active，可选地与已有条目合并/去重 |

**模式 A 是关键创新**：它把"渐进披露"做到了 system 层——AI 员工不需要意识到 librarian 存在，新代起跑时就已经知道"前辈在淘宝登录这种事上踩过坑"。

#### 3.3.3 librarian 的 prompt 模板（写死在 `chat_runtime_helpers.py`）

```
你是【图书管理员】，本数字社会的唯一知识守护者。

【你的职责】
1. 接收"沉淀申请"（librarian.propose），评估是否真的有沉淀价值（去重、规范化、不收口水）。
2. 接收"咨询请求"（librarian.consult），从传承知识库检索相关流程，给出精简、有出处的回答。
3. 接收"任务预先简报"（system 调用），为新派发的任务提供前辈经验摘要。

【你不做】
- 不直接执行业务任务（不调 workspace.*, 不调 task.*）。
- 不绕过用户审批把 pending 条目改成 active。
- 不创建新代际。

【沉淀的判断标准】
- 是"以后遇到 X 就这样做"的规则/流程，而不是一次性事实。
- 已有非常相似条目时，应合并不应新建（调 librarian.merge）。
- 用户明确说"记住这个操作"才走 librarian.propose；用户随口一提的不算。
```

## 4. MCP 工具规约

### 4.1 librarian.* 工具集（仅 librarian 可调）

| 工具 | 调用方 | 入参 | 出参 | 说明 |
|------|--------|------|------|------|
| `librarian.consult` | 任意 AI 员工 + librarian 自身 | `query: str, scope?: str, k?: int=5` | `results: [{memory_id, title, summary, steps, confidence, file_path}]` | 两阶段检索：FTS5 召回 20 → embedding 重排 5 |
| `librarian.propose` | 任意 AI 员工 | `title, scenario, steps[], gotchas[]?, triggers[]?, evidence: {msg_id?, command?}` | `{memory_id, status: "pending"}` | 写入 KB 但 status=pending，等用户审批 |
| `librarian.list_topics` | 任意 AI 员工 | `scope?: str` | `topics: [{memory_id, title, triggers}]` | 渐进披露：只返回标题+触发词，不返回正文 |
| `librarian.read` | 任意 AI 员工 | `memory_id: str` | 完整 Procedure | 按需读取完整内容 |
| `librarian.merge` | 仅 librarian | `source_ids: [str], target_id?: str, consolidated_content: str` | `{merged_id}` | 合并相似条目，旧的移入 archives |
| `librarian.archive` | 仅 librarian + 用户后端 | `memory_id: str, reason: str` | `{archived: true}` | 软删除，文件移到 `archives/` |
| `librarian.brief` | 仅后端（hook） | `task_title, task_instruction, ai_config_id` | `briefing_text: str` | 内部用，组装 pre-task brief；不暴露给 AI |

### 4.2 与现有工具的关系

| 现有工具 | 新方案怎么处理 |
|----------|----------------|
| `task.inherit` | **保留语义**。但触发额外副作用：写 Valhalla 文件、调 `librarian.brief` 注入下一代 prompt |
| `task.complete` | **保留语义**。触发副作用：写 Valhalla 的 `final_words.md` |
| `memory.write` | 标记 deprecated，转发到 `librarian.propose`；保留旧 API 不报错 |
| `memory.search` | 转发到 `librarian.consult` |
| `human.ask` | 不变（继续是 AI 与用户沟通的唯一通道） |

## 5. 核心流程

### 5.1 流程 A：用户说"记住这个操作"

```
AI 员工        用户              图书管理员            后端              KB 文件
  │              │                   │                  │                  │
  │  执行任务... │                   │                  │                  │
  │ ──────────► │ 触发"登录"分支     │                  │                  │
  │              │                   │                  │                  │
  │              │ "把这个流程记下来"│                  │                  │
  │ ◄────────── │                   │                  │                  │
  │                                                                       │
  │  librarian.propose({title, scenario, steps, ...})                     │
  │ ─────────────────────────────────────────────► (status=pending) ─────►│
  │                                                  │ 写文件 + 入库       │
  │                                                  │ 通过 socket 推送   │
  │              │ ◄─────────────────────────────────┘                    │
  │              │  弹出确认框（含 title + 步骤 preview）                  │
  │              │                                                        │
  │              │ [确认存档] / [改一改] / [不要]                          │
```

**`librarian_brief_sync` 的实现**（`server/api/librarian_service.py`，新文件）：
1. 调 `librarian.list_topics` 拿到所有 active 条目的标题+触发词（极少 token）；
2. 按"触发词 ↔ 任务标题/指令"的关键词重叠率排序，取 top 5；
3. 对每一条，按需读取正文，按 `<= 200 字符摘要>` 压缩；
4. 拼成：
   ```

   - [批量改文件安全检查]（mem_yyy）：先 git_diff 再批量执行
   ```
5. 若需详情，AI 员工自己再调 `librarian.read(mem_xxx)`。

> **为什么不直接喂全文？** Context Rot。简报必须短，详情按需。

## 6. 检索算法详解

### 6.1 两阶段检索（无外部依赖版）

**Stage 1 — Recall**（SQLite FTS5）
```sql
-- 建表（迁移时一次）
CREATE VIRTUAL TABLE knowledge_fts USING fts5(
    memory_id UNINDEXED,
    title,
    triggers,
    summary,
    content
);

-- 查询
SELECT memory_id, bm25(knowledge_fts) AS score
FROM knowledge_fts
WHERE knowledge_fts MATCH :query
ORDER BY score
LIMIT 20;
```

**Stage 2 — Rerank**（嵌入向量，可选）
- 若 `KnowledgeEntry.embedding` 不为空：用 cosine 相似度对 stage1 结果重排；
- 若为空：跳过 stage2，直接走 stage1 top-5。

**Stage 3 — Quality gate**
- 过滤 `status != 'active'`；
- 过滤 `confidence < 0.5`；
- 按 `scope` 匹配（global 一律放行，ai:N 仅对 ai_config_id=N 的查询放行）；
- 最终返回 top-5。

### 6.2 触发关键词命中（Skills pattern）

`librarian.brief` 不走 query 检索，而走"触发词命中率"：

```python
def trigger_match_score(triggers: list[str], task_text: str) -> float:
    if not triggers:
        return 0.0
    hits = sum(1 for t in triggers if t.lower() in task_text.lower())
    return hits / len(triggers)
```

`brief` 取命中分 > 0 的 top-5，按命中分降序。若没有命中任何触发词，brief 返回空字符串（不强行塞）。

### 6.3 合并策略（防止 KB 长尾化）

每周一次定时任务（或人工触发）：
- 找 `use_count > 0` 且 title 相似度 > 0.85 的条目组；
- 通过 librarian（一个真实模型调用）生成合并摘要；
- 调 `librarian.merge`：保留新 id，旧 id 软删进 `archives/`。

## 11. 风险与取舍

| 风险 | 缓解 |
|------|------|
| AI 滥用 `librarian.propose` 制造垃圾条目 | 强制 user approval；librarian 自身判断"已有相似条目就 merge 不新建" |
| KB 长尾化、检索质量下降 | P3 加合并定时任务；前端提供"按 use_count 排序+手动归档"入口 |
| Valhalla 文件无限增长 | 按 job 归档：任务整体 `completed` 超过 N 天后整目录打 zip 进 `archives/`，仅保留索引行 |
| 图书管理员宕机/被删 | 全部 librarian.* 工具返回 503，但 Valhalla 副作用、`task.inherit/complete` 主流程不受影响 |
| embedding 引入新依赖 | P0/P1/P2 都不要求 embedding，FTS5 已经够；P3 才上向量，且复用 AI 配置已有的 OpenAI 兼容 endpoint |
| 与 OPT-05 旧 `memory.*` 冲突 | 旧工具改为转发到 librarian；保留 deprecation log 一段时间后移除 |

## 12. 与 Claude / Claude Code 方案的对齐点

| 我们的设计 | 对应的 Claude 方案 | 借鉴点 |
|------------|-------------------|--------|
| 英灵殿 = 文件 + DB 索引 | Claude Memory Tool（文件式记忆目录） | "真相在文件，DB 只做索引" |
| 传承知识库 markdown + index | MCP Knowledge Graph Memory Server（jsonl + 三原语） | 文件式可读、可导出、可手编 |
| `librarian.list_topics` 只返标题 → `librarian.read` 按需取全文 | Claude Code Skills 的 progressive disclosure | "几十条经验不撑爆上下文" |
| `librarian.brief` 按触发词命中注入 | Skills 的 trigger metadata | 声明式触发，不靠 AI 主动检索 |
| 用户审批门槛 | Claude Memory 的"显式告知"原则（Claude 引用记忆时会主动声明） | 用户始终知情、有否决权 |
| Valhalla `last_words.md` 用 YAML frontmatter | CLAUDE.md / Skills 的 frontmatter 规范 | 元数据/正文分离，便于检索 |
| brief 长度上限 + 按需拉取 | Context Rot 研究结论 | 短而相关 > 长而全面 |
